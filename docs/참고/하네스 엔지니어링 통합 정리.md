# 하네스 엔지니어링 통합 정리

> 원본: `Harness Engineering Manual.txt` + `하네스 엔지니어링 완전정복.pdf` (바이브코딩 태일러, 200p, 130+ 소스)
> 정리일: 2026-03-30

---

## 0. 두 문서의 관계와 성격

**Harness Engineering Manual** (자체 제작)은 실무 운영 매뉴얼이다. "다른 저장소에서 그대로 적용할 수 있는 공용 운영 규칙"을 지향하며, 4개 레이어(헌법→상황별→자동강제→기억)와 실패 흡수 절차, 새 프로젝트 도입 체크리스트, 템플릿까지 갖춘 운용서다. 특정 프로젝트에 종속되지 않는 이식 가능한 구조를 목표로 한다.

**하네스 엔지니어링 완전정복** (바이브코딩 태일러)은 이론+사례+실습 교과서다. Mitchell Hashimoto 원문, OpenAI·Anthropic·LangChain 공식 발표, arXiv 논문, Martin Fowler 블로그까지 130개 이상의 1차 소스를 체계적으로 분석한다. TaskFlow라는 할 일 관리 웹앱을 러닝 케이스로 삼아 23챕터에 걸쳐 바이브코딩 MVP에서 프로덕션 수준의 하네스까지 점진적으로 구축하는 과정을 보여준다. 6파트 구성: 왜 지금인가 → 4개의 기둥 → 실전 설계 → 플랫폼 비교 → 운영과 최적화 → 실전 프로젝트.

**상보 관계**: Manual은 "어떻게 운영할 것인가"에 집중하고, 완전정복은 "왜 필요하고, 무엇이며, 어떻게 설계하는가"의 전체 스펙트럼을 다룬다. Manual의 4레이어 구조는 완전정복의 4기둥(Constrain·Inform·Verify·Correct)과 정확히 상보적이다. 둘을 합쳐 읽으면 하네스 엔지니어링의 이론적 근거부터 실전 도입까지 빈틈 없이 커버된다.

## 1. 핵심 테제: "모델보다 하네스가 성과를 결정한다"

### 1.1 용어의 기원

2026년 2월 5일, Terraform과 Vagrant를 만든 Mitchell Hashimoto가 블로그에 'My AI Adoption Journey'를 올리면서 '하네스 엔지니어링(Harness Engineering)'이라는 이름을 붙였다. 말(horse)의 에너지를 방향으로 바꾸는 마구(harness)처럼, AI 모델의 지능을 안정적인 결과로 바꾸는 환경을 설계하는 기술이다.

이 이름이 나온 지 2주 만에 OpenAI가 공식 채택했고, Anthropic, LangChain, Martin Fowler, Philipp Schmid가 잇달아 관련 글을 발표했다. 2026년 3월 현재 하네스 엔지니어링은 AI 개발의 가장 뜨거운 키워드가 됐다.

### 1.2 숫자로 보는 증거

| 출처 | 실험/사례 | 핵심 수치 |
|------|-----------|-----------|
| **LangChain** | 동일 GPT-5.2-Codex, 하네스만 변경 | TerminalBench 52.8% → 66.5% (+13.7%p), 순위 ~30위 → 5위 |
| **TerminalBench 2.0** | 동일 Opus 4.6 모델, 다른 하네스 | 33위 vs. 5위 (28단계 차이) |
| **OpenAI** | Codex 에이전트 프로젝트 | 100만+ 줄 프로덕션 코드, 인간 작성 0줄, 5개월, 3~7명 |
| **Stripe** | 에이전트 PR 자동 처리 | 매주 1,000건+ PR 완전 무인 자동 머지 |
| **Anthropic** | 레트로 게임 메이커 실험 | 싱글 에이전트 $9(비기능) vs. 풀 하네스 $200(완전 기능) vs. 간소화 $124.70 |
| **Gartner** | 시장 전망 | 에이전틱 AI 프로젝트 40%+ 2027년까지 중단 예측 |
| **Deloitte** | 기업 조사 | 에이전틱 AI 프로덕션 적용 기업 겨우 11% |
| **ETH Zurich** | LLM 자동생성 AGENTS.md 실험 | 성능 3% 하락, 비용 20% 증가 |
| **Manus** | 하네스 구축 과정 | 6개월 동안 5번 완전 재작성 |

핵심 통찰: 모델은 상향 평준화됐다. API 키 하나로 교체할 수 있는 모델과 달리, 하네스는 수천 시간의 엔지니어링으로만 만들 수 있다. "You can't download harnesses from Hugging Face." (Manus 팀) **하네스가 해자(moat)**다.

### 1.3 성공한 하네스의 5가지 공통 패턴

OpenAI, Stripe, Shopify, Airbnb, LangChain — 업종도 규모도 다른 기업들의 하네스에서 공통 패턴이 보인다:

1. **검증 자동화** — 모든 성공 사례에서 에이전트 출력을 자동으로 검증하는 체계가 있었다
2. **컨텍스트 정밀 관리** — 필요한 정보를, 필요한 시점에, 필요한 에이전트에게만 제공
3. **실패에서 학습** — 에이전트가 실수할 때마다 환경을 업데이트 (Hashimoto의 1실패=1규칙)
4. **점진적 신뢰 구축** — 처음부터 완전 자동화가 아니라, 보수적으로 시작해서 게이트를 열어감
5. **모델이 아니라 환경에 투자** — 더 좋은 모델을 기다리지 않고 기존 모델의 환경을 개선

## 2. 하네스의 정의와 공식

### 2.1 하네스란 무엇인가

하네스는 모델 자체가 아니라 모델 바깥에서 동작을 제어하는 운영 레이어다.

포함 요소:
- 시스템 지침: AGENTS.md, CLAUDE.md, 루트 운영 문서
- 상황별 지식: skills, playbooks, runbooks
- 도구 연결: MCP, CLI, lint/test/typecheck 명령
- 분리 실행: sub-agents, worker sessions
- 자동 검증: hooks, pre-commit checks, completion gates
- 기억과 피드백: docs, decisions, incident notes, failure review

### 2.2 Agent = Model + Harness

| 구성 | 상태 | 결과 |
|------|------|------|
| Model만 있고 Harness 없음 | API 직접 호출, 채팅만 가능 | 바이브코딩 수준에서 정체 |
| Harness만 있고 Model 없음 | 도구·규칙은 있지만 추론 능력 없음 | 규칙 기반 자동화 (기존 스크립트) |
| Model + 부분적 Harness | 도구는 있지만 검증·수정 없음 | 동작하지만 신뢰 불가 |
| **Model + 완전한 Harness** | **추론 + 도구 + 검증 + 수정** | **프로덕션 수준의 에이전트** |

Philipp Schmid(Hugging Face)의 비유: CPU=기반 모델(추론 능력), RAM=컨텍스트 윈도우(휘발성 작업 메모리), **OS=에이전트 하네스**(컨텍스트 관리, 부트 시퀀스, 라이프사이클 관리), Application=에이전트(사용자 고유 로직). 같은 CPU라도 OS가 다르면 퍼포먼스가 다르듯, 동일 LLM이라도 하네스가 다르면 결과가 완전히 다르다.

### 2.3 3개의 시대 — 누적적 진화

| 시대 | 핵심 질문 | 작업 단위 | 인간의 역할 | 문제 해결 방식 |
|------|-----------|-----------|-------------|----------------|
| 1기: 프롬프트 엔지니어링 | 어떤 말을 쓸 것인가? | 단일 API 호출 | 프롬프트 작성자 | 프롬프트를 다시 쓴다 |
| 2기: 컨텍스트 엔지니어링 | 어떤 정보가 필요한가? | 멀티턴 세션 | 정보 아키텍트 | 컨텍스트 데이터를 조정한다 |
| **3기: 하네스 엔지니어링** | **어떤 환경이 필요한가?** | **완성된 기능** | **환경 디자이너** | **도구·가드레일을 추가한다** |

결정적으로, 이 3개 시대는 **교체가 아니라 누적**이다. 하네스 엔지니어링은 프롬프트와 컨텍스트 엔지니어링을 포함하는 상위 프레임워크다.

## 3. 4대 기둥: Constrain · Inform · Verify · Correct

### 3.1 Constrain — 제한하는 기술

> "에이전트가 할 수 있는 일을 줄이면, 에이전트가 잘하는 일이 늘어난다."

해공간(solution space)을 좁히면 에이전트의 정확도가 올라간다. 직관에 어긋나는 말처럼 들리지만, "아무 글이나 써봐"보다 "500자 서평을 써봐"가 더 좋은 결과를 낳는다. 에이전트도 마찬가지다. 선택지가 적을수록 좋은 선택을 한다.

**5계층 방어 아키텍처** (OpenDev 논문, Defense-in-Depth):

| 레이어 | 기능 | 강제력 수준 |
|--------|------|-------------|
| Layer 1: 프롬프트 가드레일 | 시스템 프롬프트에 금지 행동 명시 | 부탁 수준 (소프트) |
| Layer 2: 스키마 레벨 도구 제한 | Plan 모드=읽기전용, Execute 모드=쓰기 | 구조적 제한 |
| Layer 3: 런타임 승인 시스템 | Manual / Semi-Auto / Auto 3단계 | 사용자 판단 의존 |
| Layer 4: 도구 레벨 검증 | 위험 패턴 블록리스트, stale-read 감지 | 개별 도구 안전장치 |
| Layer 5: 라이프사이클 훅 | 실행 전·중·후 커스텀 스크립트 삽입 | 물리적 차단 (가장 강력) |

**핵심 기법들**:

- **샌드박스 격리**: 마이크로VM(Firecracker, 최상 격리), gVisor(유저스페이스 커널, 10~20% 오버헤드), 강화 컨테이너(Docker + seccomp/AppArmor)
- **도구 허용 목록**: Anthropic 연구 — "Fewer tools, better outcomes. More tools or overlapping tools distract agents from efficient strategies." 도구가 많으면 에이전트가 혼란에 빠진다. 비슷한 기능의 도구 3개 → 에이전트가 선택에 토큰 낭비 + 오선택 빈도 증가. 허용 목록(allowlist) + 점진적 공개(progressive disclosure)로 해결.
- **파일 경로 제한**: 읽기 권한과 쓰기 권한 분리, .env 등 비밀 키 파일은 읽기조차 차단
- **비용 상한**: 토큰 상한(입출력 최대치) + 금액 상한(세션당/일일 한도) + 모델 티어링(계획=고성능, 구현=경제적)
- **시간/반복 제한**: 반복 횟수 상한(태스크당 50~200회), 파일 편집 감지(동일 파일 5~10회), 월클럭 타임아웃(30분~2시간)

**역설적 통찰**: "에이전트를 믿는 것"과 "에이전트를 제한하는 것"은 모순이 아니다. 제한은 불신이 아니라 설계다. 벽이 없는 도로가 위험하듯, 제약이 없는 에이전트는 위험하다.

### 3.2 Inform — 알려주는 기술

> "에이전트에게 알려주지 않은 것은, 에이전트에게 존재하지 않는 것이다."
> "동시에: 너무 많이 알려주면, 중요한 것이 묻힌다."
> 이 두 문장 사이의 균형이 Goldilocks Zone이다.

**컨텍스트 엔지니어링의 정의** (Anthropic): "바람직한 결과의 가능성을 최대화하는, 가장 작은 고품질 토큰 집합을 찾는 것." **"가장 작은"**이라는 표현에 주목. 많이 넣는 게 좋은 게 아니다. 적게, 정확하게, 필요한 시점에 넣는 것이 좋다.

**3채널 체계**:

| 채널 | 역할 | 비유 | 적재 시점 |
|------|------|------|-----------|
| 정적 문서 (AGENTS.md/CLAUDE.md) | 항상 거기 있는 지침서 | 교과서 | 세션 시작 시 자동 |
| 동적 컨텍스트 (MCP, RAG) | 실행 중 가져오는 정보 | 도서관 | 필요할 때 JIT 검색 |
| 이벤트 기반 리마인더 (Skills) | 상황에 따라 꽂히는 지침 | 시니어 개발자의 조언 | 특정 조건 충족 시 |

**좋은 Inform vs. 나쁜 Inform**:

| 차원 | 나쁜 Inform | 좋은 Inform |
|------|-------------|-------------|
| 분량 | 길고 포괄적 | 짧고 핵심적 (60줄 이하 권장) |
| 생성 방식 | LLM이 자동 생성 | 사람이 실패 경험에서 추출 |
| 구조 | 산문형 설명 | 규칙 기반, 체크리스트 형식 |
| 타이밍 | 세션 시작에 전부 로드 | 정적 + 동적 + 이벤트 조합 |
| 갱신 | 한 번 쓰고 방치 | 실패 발생 시마다 1줄 추가 |
| 검증 | "이 정도면 되겠지" | Eval로 정보 충분성 측정 |

**ETH Zurich의 경고**: LLM이 자동 생성한 AGENTS.md를 사용한 경우 태스크 성공률 3% 하락, 비용 20% 증가. LLM 생성 문서는 길고 장황하며 핵심/부수 정보의 구분이 없었다. AGENTS.md는 "에이전트가 모르는 것", 프로젝트 고유의 맥락만 담아야 한다.

**HumanLayer 데이터**: 지침 수가 늘어날수록 개별 준수율이 떨어진다. 10개 규칙 → 대부분 지킴. 50개 규칙 → 절반 무시. 100개 규칙 → 무엇이 중요한지 판단 불가.

**Goldilocks Zone** (Anthropic):

| 실패 모드 | 증상 | 결과 |
|-----------|------|------|
| 너무 구체적 (Too Brittle) | 복잡하고 하드코딩된 로직 | 취약함. 상황 조금만 바뀌면 깨짐 |
| 너무 모호 (Too Vague) | 고수준 일반 가이드라인만 | 에이전트가 제각각 행동 |
| **적정 수준 (Goldilocks)** | **행동을 안내하되 유연성 유지** | **강한 휴리스틱 + 상황 적응** |

### 3.3 Verify — 검증하는 기술

> "전통적 테스트: `assert output == expected`"
> "AI 검증: `assert output in acceptable_range`"
> 결정론적 시스템의 검증 방법으로 확률적 시스템을 검증할 수 없다.

AI 에이전트는 한 번에 200~300줄을 출력하고 한 세션에서 파일 10개를 수정한다. Microsoft Research에 따르면 코드 리뷰 효율은 300~400줄 이후 급격히 하락, 1,000줄이면 결함 발견율 사실상 0. "human eyeballing"은 에이전트 시대에 구조적으로 유지 불가능하다.

**3유형 검증 스택**:

| 유형 | 검증 대상 | 장점 | 한계 |
|------|-----------|------|------|
| 코드 실행 검증 (테스트) | 동작 여부 | 객관적, 자동화 용이 | 테스트가 없으면 무력 |
| 셀프 검증 루프 (체크리스트) | 요구사항 충족 | 추가 도구 불필요, 유연 | 에이전트 자신의 한계에 의존 |
| 외부 도구 검증 (린터/타입체커/보안스캐너) | 코드 품질·보안 | 일관된 기준 적용 | 설정·유지 비용 발생 |

한 가지만으로는 부족하다. 셋을 겹쳐야 한다.

**Eval(평가)의 기본 구조**:

```
입력(Task) → 에이전트 실행 → 출력(Result) → 평가기준(Grader) → 확률적 임계값 판정
```

TDD와의 결정적 차이: TDD에서 1+1=2가 아니면 실패, 예외 없음. Eval에서는 10번 실행해서 8번 이상 올바른 결과가 나오면 합격으로 판정. 80%라는 임계값을 정의하는 것이 Eval 설계의 감각이다.

**확률적 성공 임계값 가이드**:

| 상황 | 권장 임계값 | 이유 |
|------|-------------|------|
| 코드 포매팅, 네이밍 컨벤션 | 90%+ | 결정론적에 가까운 작업, 높은 일관성 요구 |
| 기능 구현 (CRUD, API) | 70~80% | 구현 경로가 여러 개 가능, 결과만 맞으면 OK |
| 복잡한 리팩토링 | 50~60% | 난이도 높고, 부분 성공도 가치 있음 |
| 보안 관련 (인증, 암호화) | 95%+ | 실패 1건의 비용이 극도로 높음 |

**EDD (Eval-Driven Development)**: TDD의 AI 버전. 에이전트 로직 작성 전에 Eval을 먼저 만든다. Eval이 통과할 때까지 하네스를 수정한다. "소프트웨어에서는 코드가 앱을 문서화한다. AI에서는 트레이스가 그 역할을 한다." (Arize)

### 3.4 Correct — 수정하는 기술

> "AI는 반드시 실수한다. 문제는 실수 자체가 아니라, 실수로부터 복구하는 능력이 있느냐가 문제다."

SWE-bench Verified 기준, 2026년 최고 성능 모델의 정답률이 70% 초과. 뒤집으면 30%는 틀린다. SWE-bench Pro(더 어려운 문제)에서는 최정상급 모델도 ~23% 성공률. 100% 성공률 추구는 비가 안 오는 세상을 바라는 것과 같다. 비가 올 것을 전제하고 우산을 설계하는 편이 낫다.

**4가지 복구 패턴**:

**패턴 1: 에러 리커버리** — 감지(에러 로그, 테스트 실패) → 분석(에러 파싱, 변경 이력 추적) → 복구(코드 수정, 설정 변경, 롤백). Anthropic 원칙: "Error: EPERM"보다 "Error: EPERM — /etc/hosts 파일에 대한 쓰기 권한이 없습니다. 프로젝트 디렉토리 내의 파일만 수정하세요."가 낫다. 전자를 받으면 같은 시도를 반복하지만, 후자를 받으면 접근 방식을 바꾼다.

**패턴 2: 재시도 로직** — 지수 백오프(일시적 오류: 네트워크, 레이트 리밋) + 대체 경로 시도(구현 전략 변경). 코딩 에이전트에서 "같은 방법으로 재시도"는 거의 항상 같은 실패. **Ralph Loop 패턴**: 현재 세션 종료 → 이전 실패 정보만 간결하게 정리 → 깨끗한 컨텍스트 윈도우에서 재시도. 핵심은 실패한 추론의 잔재가 컨텍스트를 오염시키지 않게 하는 것.

**패턴 3: Human-in-the-Loop** — 에스컬레이션 기준을 명시적으로 정의: 동일 에러 3회+ 재시도 실패 / 스코프 크리프 감지 / 보안 관련 결정 / 비가역적 작업 / 비용 상한 도달. 너무 빨리 넘기면 자동화의 의미 퇴색, 너무 늦게 넘기면 피해 확대.

**패턴 4: 가비지 컬렉션** — Martin Fowler 블로그에 기고한 Birgitta Bockeler(Thoughtworks)가 하네스의 세 번째 범주로 꼽은 것. 시간이 지나면 문서가 부패한다. 코드는 변경됐는데 AGENTS.md 규칙은 예전 그대로. OpenAI Codex 팀의 5번째 교훈: "Garbage collection: Periodic agent runs to identify documentation inconsistencies and constraint violations."

**Hashimoto 원칙: 실패 1건 = 방지책 1줄**

에이전트가 실수할 때마다, 그 실수를 방지하는 규칙을 AGENTS.md에 1줄 추가한다. 삭제하지 않는다. 누적시킨다. 한 달이면 6개, 석 달이면 20개, 1년이면 50개 이상. 이 파일은 프로젝트의 "학습된 지혜"가 된다. 새로운 에이전트가 투입되거나 세션이 초기화되어도 누적된 방지책은 남는다.

### 3.5 4기둥의 순환 구조

4가지는 독립이 아니라 순환한다:

```
Constrain(해공간 좁히기)
    → Inform(좁혀진 공간 안에서 방향 안내)
        → Verify(결과 검증 → 실패 시 Correct로)
            → Correct(복구 + 학습 → Constrain/Inform/Verify에 반영)
                → 사이클 완성
```

Correct에서 발견된 패턴이:
- Constrain의 새로운 규칙이 된다 (재시도 횟수 제한 → 리소스 리밋)
- Inform의 새로운 지침이 된다 ("UTC 기반 시간 처리" → AGENTS.md 추가)
- Verify의 새로운 Eval이 된다 (타임존 에러 → 자동 테스트 추가)

한 바퀴를 돌 때마다 구멍이 하나씩 막힌다. 열 바퀴를 돌면 대부분의 일반적 실패가 방지된다. 백 바퀴를 돌면, 에이전트는 6개월째 프로젝트에 참여한 시니어 개발자처럼 행동한다.

## 4. Manual의 운영 레이어 구조

### 4.1 4레이어 설계

| 레이어 | 용도 | 대표 위치 | 여기에 둘 것 | 여기에 두지 말 것 |
|--------|------|-----------|-------------|-------------------|
| **A: 헌법 문서** | 모든 작업에 불변 적용 | AGENTS.md, CLAUDE.md | 문서 우선, 테스트 필수, 재사용 원칙, 파괴 명령 금지, 완료 기준 | 긴 절차서, 조건부 분기, 특정 기능 전용, 상세 디렉터리 맵 |
| **B: 상황별 실행** | 특정 작업 유형 전용 | Skills, Playbooks, Runbooks | 문제 유형, 사용 조건, 실행 순서, 실패 대응, 검증 방법 | 범용 규칙 |
| **C: 자동 강제** | 기계적으로 위반 차단 | Hooks, pre-commit, completion gate | 사람이 잊거나 모델이 건너뛰기 쉬운 항목 | — |
| **D: 기억과 학습** | 미래 세션 재사용 | Glossary, ADR, Postmortem | 실패, 결정, 운영 사실 (규칙은 추상화, 사례는 기록) | — |

핵심 원칙: **사람 의지보다 자동화가 강하다**. 반복 위반이 발견되면 문장 규칙이 아니라 훅으로 승격한다.

### 4.2 기본 운영 원칙 10가지

1. 모든 개발은 문서와 함께 진행한다
2. 구조, 계약, 정책, API가 바뀌면 관련 문서를 즉시 갱신한다
3. 기존 패턴, 공통 모듈, 공통 용어를 우선 사용한다
4. 임시 코드, 중복 코드, 즉흥 규칙을 새 표준처럼 남기지 않는다
5. 변경 전에는 영향 범위를 확인하고, 변경 후에는 실제 영향이 반영됐는지 검증한다
6. 완료 주장은 검증 로그나 테스트 결과로 뒷받침한다
7. **실패는 수동 수정으로 끝내지 않고, 재발 방지 규칙으로 흡수한다**
8. 새로운 개념이 생기면 먼저 이름과 의미를 정의한다
9. 긴 전역 지시문보다 짧은 전역 지시문 + 필요한 순간의 세부 문서를 선호한다
10. 에이전트는 추측보다 근거를 우선하고, 사람은 예외보다 시스템 개선을 우선한다

### 4.3 실패를 하네스로 흡수하는 절차 (핵심 운영 루프)

**잘못된 대응**: 지금만 고치고 끝낸다 / 모델이 원래 그렇다고 넘긴다 / 프롬프트를 길게 덕지덕지 붙인다

**올바른 대응**:
1. 실패 유형에 이름을 붙인다
2. 왜 발생했는지 분류한다
3. 어느 레이어에서 막아야 하는지 정한다
4. 규칙, skill, hook, 문서 중 하나로 흡수한다
5. 다음 동일 작업에서 실제로 방지되는지 확인한다

**원인 분류 6가지**: 전역 규칙 부재 / 특정 작업 절차 부재 / 도구·검증 자동화 부재 / 용어·문서 불일치 / 과도한 컨텍스트 오염 / 권한·책임 경계 불명확

**흡수 대상 결정**: 모든 작업 해당 → AGENTS.md / 특정 유형만 → skill·runbook / 반복 위반 강제 차단 → hook·script / 미래 참고 중요 → docs·glossary·postmortem

## 5. AGENTS.md 작성법 — 실전 가이드

### 5.1 GitHub 2,500+ 리포 분석에서 도출된 6대 핵심 영역

1. **프로젝트 개요**: 기술 스택, 아키텍처 구조, 핵심 디렉터리 맵
2. **코딩 규약**: 네이밍 컨벤션, 포맷팅, 린터 설정, import 순서 (가장 빈번)
3. **빌드·테스트**: 빌드 명령어, 테스트 실행법, CI 파이프라인 연동
4. **금지 사항**: 절대 하면 안 되는 행동 목록 (**가장 효과가 큼**)
5. **도구 사용법**: 프로젝트 전용 스크립트, 환경 변수, DB 접속 방법
6. **알려진 함정**: 과거 실패 사례와 우회 방법, 버전 호환성 주의사항

에이전트는 해야 할 일보다 **하지 말아야 할 일을 더 정확하게 학습**한다. LLM의 사전 학습 데이터에는 다양한 코딩 스타일이 혼재하므로, "이렇게 해라"보다 "이건 하지 마라"가 더 좁고 명확한 제약으로 작동한다.

### 5.2 Before/After 10가지

| # | 나쁜 지시 (Before) | 좋은 지시 (After) |
|---|-------------------|-------------------|
| 1 | 코드를 깔끔하게 작성하세요 | 함수 20줄 이내. 변수명 camelCase. 매직넘버 금지. |
| 2 | 테스트를 충분히 작성하세요 | 모든 public 메서드에 단위 테스트. 커버리지 80% 이상. |
| 3 | 에러를 적절히 처리하세요 | HTTP API: try-catch 후 {code, message} 형태 반환. 500은 로깅 필수. |
| 4 | 보안에 신경 쓰세요 | SQL은 반드시 파라미터 바인딩. 문자열 연결 SQL 금지. |
| 5 | 성능을 고려하세요 | N+1 쿼리 금지. 목록 조회는 반드시 JOIN 또는 prefetch 사용. |
| 6 | 좋은 커밋 메시지를 쓰세요 | Conventional Commits: feat/fix/chore(scope): 50자 이내 제목. |
| 7 | 코드 리뷰 관점에서 작성하세요 | PR당 변경 파일 10개 이하. 1PR = 1기능. |
| 8 | 문서를 잘 유지하세요 | public 함수에 JSDoc 필수. @param, @returns, @throws 포함. |
| 9 | 의존성을 신중하게 관리하세요 | 새 npm 패키지 추가 시 bundle size 증가분 기록. 50KB 초과 시 대안 검토. |
| 10 | 빌드가 깨지지 않게 하세요 | 커밋 전 `npm run build && npm test` 통과 확인. CI는 Node 20 LTS 기준. |

Before의 공통점: 에이전트의 해석에 맡긴다. After의 공통점: 숫자, 도구 이름, 파일 형식 같은 측정 가능한 기준.

### 5.3 분량: 60줄의 황금률

Anthropic 권장. 이유:
- **토큰 효율**: 60줄 ≈ 800~1,200 토큰. 시스템 프롬프트의 5% 미만
- **주의력 집중**: LLM은 컨텍스트 길어질수록 초반 지시 영향력 감소. 60줄이면 전체가 '초반'
- **유지보수**: 200줄 넘으면 팀원이 읽지 않고 유지보수 포기. 60줄은 1분 안에 훑을 수 있음

60줄 넘어갈 때: 디렉토리별 AGENTS.md로 분할. 루트(전체 규칙 40줄) + src/api/(API 전용 15줄) = 55줄.

### 5.4 자가 품질 진단 10항목

| # | 점검 항목 | 기준 |
|---|-----------|------|
| 1 | 60줄 이내인가? | 넘으면 분할 필요 |
| 2 | 모든 지시가 측정 가능한가? | "깔끔하게" 같은 모호한 표현 금지 |
| 3 | 금지와 대안이 쌍으로 있는가? | "~하지 마라" 뒤에 "대신 ~해라" |
| 4 | 빌드·테스트 명령어가 있는가? | 에이전트가 복사해서 실행 가능해야 함 |
| 5 | LLM이 이미 아는 일반론은 없는가? | 프로젝트 고유 정보만 포함 |
| 6 | 최근 3개월 내 업데이트됐는가? | 오래된 규칙은 해로움 |
| 7 | 디렉토리 구조가 명시되어 있는가? | 어디에 무엇이 있는지 |
| 8 | 알려진 함정이 기록되어 있는가? | 과거 실패로부터의 학습 |
| 9 | 환경 변수 설정법이 있는가? | .env.example 경로와 필수 키 |
| 10 | git으로 버전 관리되고 있는가? | 변경 이력 추적 가능 |

7개 이상 통과하면 실전 투입 가능. 5개 미만이면 재검토 필요.

## 6. 도구 설계 패턴

### 6.1 Anthropic 7원칙

1. **고영향 워크플로우 선택** — 모든 API를 감싸지 말고 핵심 워크플로우만 도구화
2. **네임스페이스로 정리** — service_action 형식 (asana_search, asana_create)
3. **의미 있는 컨텍스트 반환** — UUID 대신 자연어 이름 포함
4. **토큰 효율 극대화** — 응답 25,000 토큰 이내, 페이지네이션/필터링 기본 제공
5. **설명문이 행동을 결정** — 사용 시점, 반환 형식, 제약 조건을 설명에 포함 (**가장 과소평가됨**)
6. **액셔너블 에러 반환** — "404 Not Found" 대신 복구 절차를 에러에 포함
7. **프로토타입에서 시작** — 프로토타입 → eval → 분석 → 설명문 개선의 반복

도구 설명문 한 줄 수정으로 올바른 도구 선택률 40% → 78%. 에이전트에게는 로직보다 설명문이 중요하다.

### 6.2 5대 오케스트레이션 패턴

| 패턴 | 구조 | 적합한 상황 | 비유 |
|------|------|-------------|------|
| **Router** | 입력 분류 → 전문 처리 경로로 분배 | 입력 유형이 명확히 구분될 때 | 안내데스크 |
| **Chain** | 단계 순차 연결, 출력→다음 입력 | 명확한 순서가 있을 때 | 자동차 조립 라인 |
| **Parallel** | 독립 작업 동시 실행 → 결과 합산 | 파일 겹침 없는 독립 태스크 | 팀원 3명 동시 작업 |
| **Orchestrator** | PM이 서브태스크 분배, 결과 통합 | 복합 기능, 인터페이스 규격 필요 | 프로젝트 매니저 |
| **Evaluator-Optimizer** | 여러 변형 생성 → 자동 평가 → 최적 선택 | 반복 사용되는 핵심 프롬프트 최적화 | A/B 테스트 |

패턴 조합의 원칙: 각 패턴이 다른 패턴의 설정 또는 입력을 결정하게 만든다. Router가 Chain의 도구와 임계값을 결정하고, Evaluator가 Chain의 재실행 여부를 결정한다.

## 7. 컨텍스트 엔지니어링 심화

### 7.1 4대 전략

**JIT(Just-in-Time) 검색**: 파일 100개 내용을 미리 로드하면 ~15,000 토큰. 경로만 유지하면 ~200 토큰. 에이전트가 실제로 필요한 파일은 보통 2~3개. JIT로 그 파일만 로드하면 3,000~4,500 토큰. **70%+ 토큰 절약**.

**하이브리드 전략**: 정적 문서(항상 필요한 것) + 동적 컨텍스트(그때그때) + 이벤트 리마인더(조건부 주입)를 조합.

**서브에이전트 위임**: 대량 탐색 결과를 서브에이전트 내부에 가두고, 부모 에이전트는 정제된 요약만 받는다. "컨텍스트 방화벽" 역할.

**적응적 컴팩션**: 컨텍스트 한계 접근 시 자동 축소. 오래된 턴 요약 → 장황한 출력 압축 → 관련 콘텐츠 그룹화 → 요약 병합 → 최저 우선순위 퇴거.

### 7.2 Context Rot 3단계 방어

| 단계 | 전략 | 기법 |
|------|------|------|
| 1단계: 예방 | Rot 시작 시점 최대한 늦추기 | JIT 검색 + Lazy Discovery + 출력 억제 + 서브에이전트 |
| 2단계: 감지 | 에이전트 행동 변화에서 징후 포착 | 반복 패턴 추적 + 루프 감지 |
| 3단계: 복구 | 컨텍스트 리셋 | 가벼운 복구(컴팩션 강제) 또는 강한 복구(Ralph Loop — 진행 파일에 상태 기록 후 새 세션) |

"에이전트가 갑자기 이상하게 행동한다"면 열에 아홉은 Context Rot다. 모델의 문제가 아니라 컨텍스트가 포화된 것이다.

### 8.2 Claude Code 핵심 — Inform의 3층 구조

1. **CLAUDE.md** (항시 로딩): 세션 시작 시 시스템 프롬프트에 자동 주입. 60줄 이내, 보편 규칙만. 디렉토리별 계층 구조 가능.
2. **Skills** (조건부 로딩): 키워드 기반 점진적 공개(Progressive Disclosure). 에이전트가 "배포" 키워드 인식 → deploy SKILL.md 활성화.
3. **MCP** (필요시 로딩): 외부 도구 표준 인터페이스. 토큰 효율적 키워드 스코어링으로 지연 탐색(lazy discovery).

이 3층 구조가 컨텍스트 효율성과 정보 풍부함 사이의 균형을 잡는다. 다른 플랫폼은 이 정도로 세분화된 Inform 계층을 갖고 있지 않다.

### 8.3 AGENTS.md vs. CLAUDE.md vs. .cursorrules

| 차원 | AGENTS.md | CLAUDE.md | .cursorrules |
|------|-----------|-----------|--------------|
| 대상 플랫폼 | 범용 (GitHub Copilot, Codex 등) | Claude Code 전용 | Cursor 전용 |
| 형식 | Markdown (자유 구조) | Markdown (자유 구조) | MDC 형식 (frontmatter + body) |
| 계층 구조 | 디렉토리별 배치 가능 | 프로젝트 + 사용자 레벨 | path 필터로 조건부 적용 |
| 플랫폼 종속성 | 낮음 (표준화 추세) | 높음 (Anthropic 전용) | 높음 (Cursor 전용) |

멀티 플랫폼 전략: AGENTS.md를 마스터 문서로 유지, CLAUDE.md와 .cursorrules는 참조 또는 심볼릭 링크. 중복 작성 피해야 불일치 방지.

## 10. Eval 파이프라인 구축

### 10.1 EDD 6단계

1. **실패 태스크 수집**: 에이전트가 실제로 틀렸던 사례를 20~50개 수집 (Anthropic 권장)
2. **카테고리화**: 타입 안전성, API 규격, 엣지 케이스, 코딩 컨벤션, 테스트 커버리지 등
3. **자동 실행 환경 구축**: 태스크마다 깨끗한 Git 브랜치 → 에이전트 실행 → 채점
4. **채점 함수 작성**: 코드 기반(결정론적) + LLM 기반(정성적) + 혼합
5. **기준선 측정**: 20개 태스크 x 10회 실행 = 200회 결과 → 카테고리별 통과율 기록
6. **반복 개선**: 가장 낮은 카테고리부터 하네스 수정 → 재측정 → 수치 변화 확인

실전 사례: TaskFlow에서 CLAUDE.md 3줄 추가 → 통과율 60% → 80% (+20%p)

### 10.2 CI/CD 통합

- CLAUDE.md, .claude/ 디렉토리, eval/ 디렉토리 변경 시에만 Eval 실행 (비용 절약)
- regression-tolerance 0.05: 기준선 대비 5%p 이상 하락하면 PR 실패 처리
- 하네스 수정이 의도치 않은 부작용을 일으키는지 자동 탐지

## 11. 비용 관리

### 11.1 토큰 경제학

| 토큰 유형 | Claude Opus (MTok) | Claude Sonnet (MTok) | Claude Haiku (MTok) |
|-----------|--------------------|-----------------------|---------------------|
| 입력 | $15 | $3 | $0.80 |
| **출력** | **$75** | **$15** | **$4** |
| 캐시 읽기 | $1.50 | $0.30 | $0.08 |

두 가지 핵심:
1. **출력 토큰이 입력보다 5배 비싸다** — "간결하게 응답하라"가 직접적 비용 절감
2. **캐시 읽기는 입력의 10%** — 시스템 프롬프트는 두 번째 호출부터 90% 절감

### 11.2 모델 라우팅

| 작업 유형 | 권장 모델 | 비용 배수 (Opus=1) |
|-----------|-----------|---------------------|
| 아키텍처 설계, 복잡한 버그 | Opus | 1x |
| 기능 구현, 테스트 작성 | Sonnet | 0.2x |
| 포맷팅, 단순 변환 | Haiku | 0.05x |

전체 작업의 70%는 Sonnet이나 Haiku로 처리 가능. Opus를 30% 핵심 작업에만 집중 투입하면 전체 비용 40~60% 감소.

### 11.3 Reasoning Sandwich

LangChain이 TerminalBench 개선에 사용한 기법:
- **계획 단계**: 고추론 (비싼 모델, 높은 추론 예산)
- **구현 단계**: 표준 추론 (경제적 모델)
- **검증 단계**: 고추론 (다시 비싼 모델)

양 끝에 비싼 추론을 배치하고 가운데는 경량화. 계획이 정확하면 구현 난이도가 낮아지고, 검증이 정확하면 구현의 실수를 잡아낸다.

### 11.4 비용 절감 5기법

1. **컨텍스트 효율적 도구 설계**: 빌드 성공 시 "Build succeeded"만 반환, 실패 시에만 에러 전문
2. **지연 도구 탐색**: MCP 도구 스키마를 전부 로딩하지 말고 키워드 기반으로 필요한 것만
3. **반복 속도 최적화**: 긴 프롬프트 한 번보다 짧은 피드백 루프 여러 번이 총 토큰 소비 적을 수 있음
4. **태스크별 토큰 예산**: 버그 수정 50K, 기능 구현 200K, 대규모 리팩토링 500K
5. **이터레이션 캡**: 파일 수정 횟수 추적, 과도한 반복 시 접근 재검토 시그널

## 12. 실패하는 하네스의 7가지 징후

### 설계 안티패턴

| 징후 | 증상 | 진단 방법 | 대응 |
|------|------|-----------|------|
| **1. 과도한 제약** | 승인 대기에 시간 50%+ 소비 | 에이전트 작업 시간 중 실제 구현 비율 측정 | 저위험 작업부터 자동 승인 전환 |

| **3. Eval 부재** | "감으로 하네스 수정", 수치 증명 불가 | 하네스 수정 효과를 수치로 말할 수 있는가? | 20개 실패 태스크로 기준선 측정부터 시작 |
| **4. 모델 의존 착각** | "다음 모델이 나오면 해결될 것" | 팀 회의에서 "다음 모델이 나오면" 빈도 | 현재 모델에서 하네스 개선 여지를 먼저 측정 |

### 운영 안티패턴

| 징후 | 증상 | 진단 방법 | 대응 |
|------|------|-----------|------|
| **5. 하네스 부패** | 문서와 코드의 괴리 | CLAUDE.md가 실제 코드와 일치하는가? | 주기적 garbage collection 에이전트 운영 |
| **6. 비용 블랙홀** | 태스크별 비용 분해 불가 | "상위 3개 비용 태스크 유형은?"에 즉시 답할 수 있는가? | 모든 LLM 호출에 태스크 ID+비용 로깅 |
| **7. 거버넌스 부재** | 하네스 변경의 리뷰 프로세스 없음 | CLAUDE.md 마지막 수정자와 이유를 Git에서 확인 가능한가? | 하네스 변경을 코드 변경과 동일한 PR 프로세스로 관리 |

**우선순위**: 가장 먼저 해결할 것은 **3번(Eval 부재)** → 나머지 6개의 개선 여부를 판단할 수 없기 때문. 두 번째는 **5번(하네스 부패)** → 부패한 토대 위에 좋은 건물을 올릴 수 없음.

### 하네스 건강도 자가진단 7문항

3개 이상 "아니오"면 하네스가 쇠퇴 경로에 있을 가능성 높음:

1. 에이전트 작업 시간의 70% 이상이 실제 구현에 쓰이는가?
2. CLAUDE.md/AGENTS.md가 100줄 이내인가?
3. 하네스 수정의 효과를 수치로 증명할 수 있는가?
4. 현재 모델에서 하네스 개선 여지를 측정했는가?
5. CLAUDE.md가 실제 코드와 일치하는가?
6. 태스크 유형별 에이전트 비용을 알고 있는가?
7. 하네스 변경에 리뷰 프로세스가 있는가?

## 13. 서브에이전트 운영 원칙 (Manual)

서브에이전트는 역할놀이가 아니라 **context firewall**이다.

**언제 쓰는가**: 큰 검색 결과가 메인 스레드를 오염시킬 때 / 독립적인 조사·검증·리뷰 작업 / 구현과 검증을 병렬 처리

**언제 쓰지 않는가**: 지금 당장 결과가 필요한 핵심 작업 / 아주 작은 단일 파일 수정 / 읽지 않고 떠넘기는 탐색

**원칙**:
1. 메인 스레드는 방향과 통합을 맡는다
2. 서브에이전트는 좁고 검증 가능한 일만 맡는다
3. 서브에이전트 결과는 요약본만 상향 전달한다
4. 같은 실수 패턴이 반복되면 sub-agent 추가보다 harness 개선을 먼저 본다

## 15. 금지 패턴 (Anti-patterns)

Manual과 완전정복에서 공통으로 경고하는 것들:

1. 실패할 때마다 전역 프롬프트만 길게 늘리는 것
2. 사람이 늘 기억할 거라고 가정하는 것
3. 에이전트가 디렉터리 구조를 모른다고 문서에 트리 전체를 붙이는 것
4. 작은 문제를 새 추상화/새 의존성으로 덮는 것
5. 문서와 실제 운영 계약을 분리해 두는 것
6. 서브에이전트를 역할 장식처럼 남용하는 것
7. 검증 없이 "완료"라고 보고하는 것

## 16. 유지보수: 하네스는 정원이다

하네스는 계속 커지면 오히려 성능을 떨어뜨린다. "하네스는 코드보다 빠르게 썩는다."

### 정기 정리 기준
- 최근 8주 이상 한 번도 필요 없던 규칙
- 같은 내용의 중복 규칙
- 원래 문제를 더 이상 일으키지 않는 임시 규칙
- 전역에 있을 이유가 없는 상세 규칙

### 비대화 경고 신호
- 에이전트가 규칙을 자주 충돌시키며 해석한다
- 전역 문서가 지나치게 길다
- 같은 규칙이 여러 문서에 복제돼 있다
- tool 선택이나 절차 선택에 시간을 낭비한다
- 완료 전에 꼭 필요한 검증을 여전히 자주 빠뜨린다

### 대응
1. 전역 규칙을 줄인다
2. 조건부 규칙을 skills로 분리한다
3. 반복 체크를 hooks로 옮긴다
4. 오래된 문서를 archive로 내린다

## 17. 두 문서의 결론이 수렴하는 지점

**Manual의 한 줄 요약**:
> "좋은 harness는 '에이전트가 잘하길 기대하는 문장 모음'이 아니라, 실패를 규칙·문서·skill·hook으로 누적 흡수하는 운영 시스템이다."

**완전정복의 한 줄 요약**:
> "하네스는 코드보다 빠르게 썩는다. 코드에 CI/CD가 필요한 것처럼 하네스에도 지속적 검증이 필요하다."

**공통 핵심**: 하네스는 한 번 만들고 끝나는 산출물이 아니라, 실패로부터 학습하며 계속 진화하는 **살아있는 시스템**이다.

바이브코딩이 입문이라면, 하네스 엔지니어링은 목적지다. 바이브코딩을 버리는 게 아니라, 그 위에 검증 레이어를 쌓는 것이다. AGENTS.md 한 장을 작성하는 것부터 시작할 수 있다.

> "모델을 바꾸기 전에, 실패가 다시 발생하지 않도록 harness를 바꾼다."
> — 두 문서가 공유하는 핵심 원칙
