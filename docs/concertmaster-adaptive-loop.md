# Concertmaster Adaptive Loop — 전략 설계 문서

> "We adapt to how you decide — not to make you comfortable, but to make your thinking better."

## 1. 문제 정의

Overture의 Concertmaster(악장)는 현재 **읽기 전용 코칭 레이어**다. 기존 분석 인프라의 출력을 통합하여 사용자에게 메타적 인사이트와 단계별 코칭을 보여준다. 세션 수에 따라 tier가 올라가고, 더 많은 정보가 노출된다.

하지만 이 정보는 **보여주기만** 한다. 프로세스 자체에는 반영되지 않는다.

GPT는 사용자의 대화 패턴을 숨겨진 프로파일로 구축하여 응답 수준을 조절한다. 사용자는 이걸 모른다. 이건 **채팅 경험 최적화**다.

Overture가 할 수 있는 것은 근본적으로 다르다: **의사결정 패턴을 수집하여 판단 품질 자체를 향상시키는 적응형 피드백 루프**.

---

## 2. 철학: Overture의 적응은 무엇을 위한 것인가

### 세 가지 적응 모델의 비교

| | GPT | Netflix | Overture |
|---|---|---|---|
| **적응 대상** | 말하는 방식 | 좋아하는 것 | 판단하는 방식 |
| **목적** | 대화 경험 최적화 | 체류 시간 극대화 | 판단 품질 향상 |
| **성공 기준** | 사용자 만족도 | 시청 시간 | 더 나은 의사결정 |
| **투명성** | 숨김 | 부분적 | 투명 + 사용자 통제 |
| **데이터 귀속** | 플랫폼 | 플랫폼 | 사용자 |
| **필터 버블 위험** | 낮음 (대화는 일회성) | 높음 | 관리 필요 (아래 참조) |

### 핵심 차별화

GPT는 "이 사람이 어떻게 말하는지"를 프로파일링한다.
Overture는 "이 사람이 어떻게 판단하는지"를 프로파일링한다.

전자는 대화 텍스트에서 추출한다 — 어휘 수준, 응답 길이 선호, 톤.
후자는 의사결정 행위에서 추출한다 — 무엇을 수용하고, 무엇을 거부하고, 어디서 멈추고, 무엇을 놓치는가.

이건 근본적으로 다른 종류의 데이터이고, 근본적으로 다른 종류의 가치를 만든다.

## 3. 세 가지 핵심 설계 질문

### Q1. 적응의 범위 — 어디까지 바꿀 것인가

적응에는 네 개의 층위가 있다. 각각의 데이터 요구량과 위험도가 다르다.

#### Layer 1: 톤/표면 적응
GPT가 하는 것. "이 사람은 간결하게 말하니까 간결하게 답하자."
Overture에서 이건 **가장 가치가 낮다**. Overture는 대화 제품이 아니라 판단 프로세스 제품이기 때문이다. 말투를 맞추는 건 기본 예의지, 핵심 가치가 아니다.

#### Layer 2: 프로세스 파라미터 적응
같은 구조, 다른 세팅:
- 페르소나 수 조절 (2개 vs 5개)
- Reframe 깊이 (표면 가정 vs 인식론적 도전)
- Refine 권장 반복 횟수
- 어떤 증거를 우선시할지 (정량 vs 정성)

방법론을 존중하면서 튜닝하는 것. 위험도 낮고 가치 높다. 5-10회 의사결정 데이터면 작동 가능.

#### Layer 3: 프로세스 구조 적응
스텝 자체의 순서나 존재를 바꾸는 것:
- 숙련자는 Decompose를 건너뛰고 Reframe으로 직행
- 어떤 사용자는 Rehearse가 3라운드 필요하고, 어떤 사용자는 Refine이 더 필요
- 시간 압박이 항상 있는 사용자 → 파이프라인 압축

**위험**: 사용자가 실제로 필요한 단계를 건너뛸 수 있음. 높은 확신 필요.

#### Layer 4: 인지 개입 — Overture만의 영역
사용자 패턴에 **반대로** 밀어주는 것:
- "최근 5개 의사결정에서 재무 리스크를 일관되게 과소평가하셨습니다 — 이 페르소나가 보완합니다"
- "첫 reframe을 항상 수락하시는 경향이 있습니다 — 대안 2개를 더 준비했습니다"
- "당신은 보통 3회 반복하는데 이번엔 1회에 끝냈습니다. 의도적인 건가요?"

**이것이 Overture를 근본적으로 다르게 만드는 층이다.** GPT는 맞춰준다. 좋은 의사결정 어드바이저는 필요한 곳에서 밀어준다.

#### 결론: 점진적 확대

```
첫 사용       → Layer 1 (기본 톤 적응)
5회 사용 후   → Layer 2 (파라미터 튜닝)
15회 사용 후  → Layer 3 (구조 제안, 선택적)
항상, opt-in  → Layer 4 (인지 개입)
```

Layer 4가 가장 가치 있지만, Layer 2부터 시작하는 것이 현실적이다. Layer 4는 가장 먼저 구현될 수도 있다 — 현재 Concertmaster의 coaching 메시지가 이미 이 방향이기 때문이다.

### Q2. 투명성의 정도 — 보여주면 도움이 될까, 방어적으로 만들까

#### "당신의 패턴입니다" 페이지가 위험한 이유

치료사 비유가 정확하다. 좋은 치료사는 "당신은 갈등을 회피하는 패턴이 있습니다"라고 선언하지 않는다. "방금 그 부분을 빨리 넘기셨는데, 거기에 뭐가 있었을까요?"라고 묻는다.

전자는 **분류당하는 느낌**을 준다. 후자는 **발견하는 느낌**을 준다.

프로필 페이지를 만들면:
- **Hawthorne 효과**: 관찰당한다는 인식 → 행동 변경. "reframe을 너무 빨리 수락한다"고 보여주면, 사용자는 내용과 무관하게 거부하기 시작할 수 있다
- **Performance anxiety**: 의사결정 도구가 오히려 의사결정을 왜곡한다
- **분류 저항**: "나를 이런 유형으로 넣지 마" — 특히 수준 높은 고민을 중시하는 사용자에게서 더 강하다

#### 대안: 판단 시점의 맥락적 삽입

프로필 페이지 대신, **각 스텝에서 필요한 순간에 인사이트를 넣는다:**

**Rehearse 페르소나 선택 시:**
> "이전 의사결정들에서 규제/법무 관점이 포함되지 않았던 경우가 많습니다. 추가해볼까요?"

**Refine에서 1회 반복 후 완료하려 할 때:**
> "이전에 2회 이상 반복했던 의사결정들이 실행 단계에서 더 적은 수정을 겪었습니다."

**Reframe에서 첫 번째 결과를 바로 수락할 때:**
> "이전 분석들 중 대안을 검토한 경우의 품질 점수가 평균 18% 높았습니다."

이 접근의 장점:
- **맥락적** — 지금 이 판단에 관련됨
- **실행 가능** — 즉시 행동으로 연결됨
- **비판단적** — "당신의 약점"이 아니라 "이런 옵션도 있습니다"
- **근거 투명** — "이전 의사결정들에서" 기반을 밝힘

#### 세 가지 정보 공개 수준

| 수준 | 내용 | 공개 방식 |
|------|------|----------|
| **행동 사실** | "15회 의사결정 완료, 평균 Reframe 4분" | 항상 (대시보드) |
| **패턴 관찰** | "최근 8회 중 7회에서 첫 reframe 수락" | 요청 시 + 대시보드 |
| **해석적 제안** | "이 패턴은 대안 탐색 부족을 시사합니다" | 맥락적 삽입 (판단 시점에만) |

**핵심 원칙: 사용자가 자기 데이터를 "발견"하게 만들어라. "선언"하지 마라.**

Quantified Self(피트니스 트래커)가 작동하는 이유는 사용자가 숫자를 보고 자기가 해석하기 때문이다. "걸음 수: 3,200"은 판단이 아니라 사실이다. 해석("운동 부족이시네요")은 사용자 몫이다.

Overture도 같은 접근: **사실을 보여주고, 해석은 사용자에게 맡기되, 프로세스 안에서 부드럽게 넛지한다.**

### Q3. 차별화 vs 편향 강화 — 가장 어려운 질문

#### 문제의 본질

모든 개인화는 필터 버블을 만든다. Netflix는 이미 좋아하는 것을 추천하고, Spotify는 이미 듣는 것을 재생한다. 엔터테인먼트에서는 괜찮지만, **의사결정에서 필터 버블은 재앙**이다.

Overture가 "이 사용자는 빠르고 대담한 결정을 선호합니다"를 학습한 후:
- 경고성 페르소나를 줄이고
- 반복 주기를 단축하고
- Reframe에서 도전적 대안을 약화시키면

사용자는 **더 편안하게 느끼면서 더 나쁜 결정을 내리게 된다.**

#### 좋은 코치 모델

좋은 스포츠 코치가 하는 네 가지:

1. **관찰 (Observe)**: 자연스러운 경향 파악
2. **활용 (Leverage)**: 강점을 극대화 — 이미 잘하는 것에 맞춤
3. **보완 (Compensate)**: 약점에 필요한 것을 추가
4. **도전 (Challenge)**: 성장을 위해 적절히 밀어줌

대부분의 개인화 시스템은 1+2만 한다. **Overture는 네 가지를 모두 해야 한다.**

#### 비율: Comfortable Discomfort

```
70% Leverage   — 사용자 스타일에 맞게 (생산적 느낌)
20% Compensate — 사용자가 놓칠 것을 추가 (지원받는 느낌)
10% Challenge  — 성장을 자극 (확장되는 느낌)
```

항상 편안하면 → 필터 버블.
항상 도전받으면 → 사용 중단.

이 비율은 고정이 아니다. 의사결정의 중요도에 따라 조절된다 — 중요한 결정일수록 Challenge 비율을 높이는 것이 자연스럽다.

#### "약점"을 누가 정의하는가

"이 사용자가 reframe을 빨리 수락하는 건 약점인가 강점(효율성)인가?"

시스템이 일방적으로 정의하면 오만하다. 네 가지 판단 기준을 조합한다:

**기준 ①: 연구 기반선 비교**
40편 논문에서 도출된 "좋은 의사결정" 패턴과 비교.
- 예: "대안 최소 3개 검토가 결과 품질과 상관관계" → 항상 1개만 보면 플래그
- 장점: 객관적 근거
- 한계: 연구는 평균이지, 이 사람에게 맞는 건 아닐 수 있음

**기준 ②: 결과 데이터 비교 (gold standard)**
이 사용자의 과거 결정 중, 어떤 패턴일 때 결과가 좋았는지 추적.
- 예: "3회 이상 반복한 결정이 실행 후 수정 빈도가 40% 낮았음"
- 장점: 이 사람에게 실제로 맞는 근거
- 한계: 데이터 축적에 시간 필요, 결과 추적 어려움

**기준 ③: 자기 분산(variance) 비교**
이 사람의 평소와 다른 행동을 플래그.
- 예: "보통 3회 반복하는데 이번엔 1회에 끝냈습니다"
- 장점: 절대적 기준 없이 인식 제공
- 한계: 평소 패턴이 나쁘면 편향 고착

**기준 ④: 사용자 선언 목표 (가장 존중적)**
사용자가 직접 말한다: "나는 대안을 더 많이 보고 싶다" / "나는 갈등 회피 경향이 있는 것 같다"
- 시스템은 **그들이 되고자 하는 모습**에 맞춰 도움
- "우리가 당신에게 뭐가 좋은지 안다"가 아니라 "당신이 원하는 방향으로 가도록 돕겠다"

**실제 구현에서는 ①+③을 기본, ②를 점진적 추가, ④를 선택적 강화로 조합한다.**

#### 절대 하지 말 것: Matthew Effect

> "Sophistication Index가 높은 사용자에게 더 깊은 분석을 제공"

이건 GPT가 암묵적으로 하는 것이다. 이미 잘 생각하는 사람에게 더 좋은 지원을 주고, 도움이 가장 필요한 사람에게는 덜 주는 **Matthew Effect**.

Overture는 **반대**여야 한다: 판단 패턴에서 보완이 필요한 사용자에게 더 많은 넛지를, 이미 균형 잡힌 사용자에게는 효율적 프로세스를.

## 4. 유의미한 데이터: 무엇을 수집할 것인가

### 핵심 원칙: 프로세스가 곧 데이터

사용자에게 추가 입력을 요구하지 않는다. Overture의 4단계 프로세스를 사용하는 행위 자체에서 모든 핵심 시그널이 추출된다.

### High-Signal 데이터 (판단 품질을 직접 드러냄)

| 시그널 | 측정 방법 | 수집 단계 | 추가 입력 |
|--------|----------|----------|----------|
| **Reframe 수용 패턴** | 수락/수정/거부 비율 | Reframe | 불필요 (클릭) |
| **가정 검토 깊이** | 가정별 평가 여부, 평가 분포 | Reframe | 불필요 (클릭) |
| **페르소나 피드백 반응** | 어떤 피드백에 시간을 쓰고 어떤 걸 넘기는지 | Rehearse | 불필요 (체류시간) |
| **페르소나 선택 편향** | 어떤 유형을 선택/제외하는지 | Rehearse | 불필요 (클릭) |
| **수렴 행동** | 반복 횟수, 반복 간 변화량 | Refine | 불필요 (자동 측정) |
| **편집 패턴** | 생성 내용 중 수정한 부분 vs 그대로 둔 부분 | 전체 | 불필요 (diff 추적) |
| **역할 배분 성향** | AI vs 사람 역할 할당 비율 | Recast | 불필요 (클릭) |

### Medium-Signal 데이터 (집적하면 유용)

| 시그널 | 비고 |
|--------|------|
| 스텝별 소요 시간 | 어디서 멈추고 어디를 빨리 넘기는지 |
| 재방문 패턴 | 완료 후 돌아오는지, 언제 돌아오는지 |
| 전략 반복 | 같은 reframing 전략만 반복 사용하는지 |
| 교차 결정 일관성 | 이전 결정에서 플래그된 패턴이 다음에도 나타나는지 |

### Low-Friction 추가 입력 (선택적)

- **결정 완료 시**: 단일 슬라이더 — "이 결정에 얼마나 확신하시나요?" (1-5)
- **결과 추적**: 1-2주 후 알림 — "이 결정이 어떻게 진행되고 있나요?" (👍/👎/🤔 3버튼)
- 이 두 개만 있으면 outcome tracking의 기초가 만들어진다

### 추가하지 말 것

- 매 결정 후 자유 텍스트 성찰 프롬프트
- 다차원 평가 스케일
- 필수 설문/평가

## 5. Cold Start 문제와 데모의 새로운 역할

### 문제

적응형 루프의 가치를 체감하려면 5-15회의 의사결정 데이터가 필요하다. 하지만 대부분의 사용자는 1-2회 사용 후 이탈한다. 전형적인 chicken-and-egg:

```
제품이 사용할수록 좋아진다 → 하지만 사용자가 충분히 쓰지 않는다
```

### 현재 데모의 한계

현재 `DemoWalkthrough.tsx`는:
- **하나의 고정 시나리오** (CEO의 AI 도입 지시)
- **격리된 환경** — signal-recorder와 연결 안 됨, 데이터 미보존
- **기능 소개 목적** — "이런 기능이 있습니다"를 보여줌
- 사용자의 데모 내 선택이 `demoChoices` 객체에 저장되지만 세션이 끝나면 사라짐

### 새로운 역할: 데모를 "첫 번째 학습 데이터"로 전환

데모의 목적을 **기능 소개**에서 **패턴 학습 + 시드 데이터 수집**으로 전환한다.

#### 패턴 학습이란

사용자가 데모에서 배우는 것은 특정 AI 도입 사례에 대한 답이 아니다. **전이 가능한 판단 메타스킬**이다:

| 데모에서 하는 것 | 학습되는 메타패턴 |
|-----------------|------------------|
| 전제를 의심해서 질문이 바뀌는 걸 봄 | "가정이 결과를 지배한다" |
| 역할을 재배치해서 리스크가 바뀌는 걸 봄 | "실행 설계가 리스크를 결정한다" |
| 페르소나가 내가 놓친 걸 짚는 걸 봄 | "다양한 관점이 맹점을 드러낸다" |
| 2회 반복으로 수렴되는 걸 봄 | "반복이 수렴을 만든다" |

#### LLM 가변성 문제와 해결

LLM은 매번 다른 출력을 준다. 따라서 "이 시나리오에서 이런 패턴을 보세요"라고 가르치기 어렵다.

**해결: 데모 시나리오는 사전 설정(pre-set)으로 운영한다.**

현재 데모가 이미 이 구조다 — LLM을 호출하지 않고 사전에 준비된 콘텐츠를 보여준다. 이걸 의도적으로 유지하되, 교육 효과를 극대화하도록 콘텐츠를 설계한다.

핵심은 **사용자의 반응은 실시간**이라는 것이다. 콘텐츠는 고정이지만, 사용자가 어떤 전제를 의심하고, 어떤 역할을 재배치하고, 어떤 피드백에 반응하는지는 그 사람만의 진짜 데이터다.

#### 데모에서 시드 데이터 수집

현재 데모의 `demoChoices` 구조:
```typescript
// 이미 추적 중인 것
demoChoices.doubtedPremises  // Set<number> — 어떤 전제를 의심했는지
demoChoices.actorSwaps       // Map<string, string> — 역할 재배치
```

이걸 signal-recorder로 연결하면:

```
데모 완료 시 수집 가능한 시드 데이터:
├── 가정 의심 성향: 3개 전제 중 몇 개를 의심했는가 (0-3)
├── 역할 배분 성향: AI에 얼마나 위임했는가
├── 페르소나 피드백 반응: 어느 관점에 더 오래 머물렀는가
└── 수렴 인내심: 2단계 반복을 끝까지 봤는가 vs 스킵했는가
```

이 시드 데이터만으로도 Concertmaster의 첫 번째 프로필을 만들 수 있다:
- **가정 의심 0개** → 첫 실제 Reframe에서 "전제를 하나씩 눌러보세요" 넛지
- **AI 위임 100%** → 첫 실제 Recast에서 "체크포인트를 충분히 두세요" 넛지
- **피드백 스킵** → 첫 실제 Rehearse에서 특정 관점 자동 추천

#### 다중 데모 시나리오의 가능성

하나의 시나리오가 아니라, **각각 다른 패턴을 교육하는 2-3개의 시나리오**:

| 시나리오 | 핵심 교훈 | 수집하는 시그널 |
|---------|----------|---------------|
| **CEO의 AI 도입** (현재) | 가정 의심의 파괴력 | 가정 의심 성향 |
| **B2B 가격 전략** (신규) | 이해관계자 관점의 가치 | 페르소나 반응 패턴 |
| **제품 리디자인** (신규) | 반복 수렴의 효과 | 수렴 인내심 |

사용자가 하나만 해도 시드 데이터가 생기고, 셋 다 하면 꽤 풍부한 초기 프로필이 만들어진다.

#### 사용자 동의와 커뮤니케이션

이 데이터 수집은 **사용자에게 이익**이 되므로 투명하게 공지한다:

> "데모를 진행하면서 당신의 판단 성향을 파악합니다. 이 데이터는 향후 분석에서 더 정교한 코칭을 제공하는 데 사용됩니다."

이건 숨길 필요가 없다 — Overture의 차별화 포인트 자체가 이것이기 때문이다. GPT가 숨기는 것을 Overture는 기능으로 내세운다.

### 전체 흐름

```
┌─────────────────────────────────────────────────────┐
│                    DATA COLLECTION                   │
│                                                      │
│  Demo (시드)  →  실제 사용 (점진적 축적)              │
│  ↓ demoChoices     ↓ signal-recorder                 │
│  ↓ 체류시간        ↓ eval-engine                     │
│                    ↓ judgment records                 │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│                  PATTERN INFERENCE                   │
│                                                      │

└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│                APPLICATION (피드백)                   │
│                                                      │
│  A. 프롬프트 주입 (LLM 호출 시 사용자 패턴 반영)       │

│                                                      │
│  B. UI 넛지 (판단 시점에 맥락적 제안)                  │

│       품질 점수가 평균 18% 높았습니다."                 │
│                                                      │
│  C. 페르소나 자동 추천 (blind spot 보완)               │

│                                                      │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
              다음 의사결정에 반영
                    (반복)
```

### 현재 인프라와의 매핑

| 적응형 루프 컴포넌트 | 현재 존재하는 인프라 | 필요한 확장 |
|--------------------|--------------------|-----------|
| Data Collection | `signal-recorder.ts`, `eval-engine.ts` | 데모 연결, 체류시간 추적, 편집 diff |
| Pattern Inference | `analyzeStrategyPerformance()`, `getWorstPerformingEvals()` | 교차 결정 패턴 분석, 연구 기반선 비교 |
| Prompt Injection | `context-chain.ts`의 `inject*Context()` | 사용자 패턴을 프롬프트에 주입하는 레이어 추가 |
| UI Nudge | `concertmaster.ts`의 `getStepCoaching()` | 교차 결정 데이터 기반 코칭 메시지 |
| Persona Recommendation | 없음 | blind spot 기반 페르소나 자동 추천 |
| Dashboard | `ConcertmasterStrip.tsx` | 패턴 시각화, 트렌드 차트 |

## 7. 안전 장치

### 편향 강화 방지

적응형 루프가 사용자의 기존 편향을 강화하지 않도록:

1. **70-20-10 비율 모니터링**: leverage/compensate/challenge 비율이 극단으로 치우치면 자동 조정
2. **연구 기반선 하한선**: 논문에서 도출된 최소 기준 이하로는 프로세스를 단축하지 않음
3. **다양성 강제**: 페르소나 선택에서 최소 1개는 사용자의 일반적 선호와 다른 관점 포함
4. **자기 분산 알림**: 사용자의 평소 패턴과 크게 다른 행동 시 의식적 확인 유도

### 프라이버시

1. **데이터는 사용자 소유**: localStorage 기본, Supabase는 사용자 계정에만 귀속
2. **투명한 수집**: 무엇을 수집하는지 언제든 확인 가능
3. **삭제 가능**: 사용자가 프로필 데이터를 언제든 초기화 가능
4. **해석 불가 데이터 미수집**: 분석할 수 없거나 의미 없는 데이터는 수집하지 않음

## 8. 해자(Moat)로서의 가치

이 적응형 루프가 만드는 해자는 세 가지 층으로 구성된다:

### 1층: 방법론 해자 (복제 가능, 하지만 어려움)
- 40편 논문 기반 eval 시스템
- 4단계 프로세스 설계
- 연구-코드 매핑

### 2층: 데이터 해자 (복제 불가능)
- 개별 사용자의 15-50회 의사결정 패턴
- 전략별 성과 데이터
- 결과 추적 상관관계

### 3층: 학습 해자 (시간이 곧 해자)
- 사용자가 Overture를 통해 형성한 메타인지 습관
- 자기 판단 패턴에 대한 인식
- 도구와의 co-evolution

**2층과 3층은 경쟁자가 복제할 수 없다.** 사용자가 Overture에서 20번의 의사결정을 내린 후 축적된 데이터와 학습은, 다른 도구로 이전할 수 없는 자산이다.

## 10. 보충: "보여주기 레이어"의 역할과 적응형 루프와의 접점

> 이하 내용은 2026-03-27 세션에서 추가. 보상/메타인지 요소를 구현한 후, 이 문서의 설계와 대조하며 정리한 관찰.

### 10.1 현재 구현된 "보여주기 레이어" — Application 계층의 B+D

이 문서의 6절 다이어그램에서 APPLICATION 계층은 네 가지 경로를 정의한다:

```
A. 프롬프트 주입 (LLM 호출 시 사용자 패턴 반영)     ← 미구현
B. UI 넛지 (판단 시점에 맥락적 제안)                  ← 부분 구현 ★
C. 페르소나 자동 추천 (blind spot 보완)               ← 미구현
D. 주기적 인사이트 (대시보드, Concertmaster strip)     ← 부분 구현 ★
```

2026-03-27에 구현한 것:

| 구현 | 위치 | 경로 | 비고 |
|------|------|------|------|
| 첫 사용 counterfactual 팁 | concertmaster + plugin | B | 데이터 없이도 작동 (pre-data value) |
| 성장 추적 (assumption 수, persona 정확도, DQ 원소) | concertmaster | D | 교차 결정 비교, 하지만 아직 단일 메트릭 기준 |
| 마이크로 인정 (earned only) | plugin SKILL.md | B | "침묵 > 범용 칭찬" 원칙 |
| DQ 질적 해석 (강점/약점 귀인) | RefinementLoopStep | D | 첫 사용은 counterfactual, 재사용은 원인 귀인 |
| tone 분류 UI (neutral/positive/counterfactual) | ConcertmasterInline | B+D | 시각적 구분 (골드/초록/파랑) |

이것들은 Application 계층의 **출력 표면(output surface)**이다. 적응형 루프가 완성되면, 같은 UI 컴포넌트가 데이터 기반 인사이트를 보여주게 된다. 지금은 하드코딩된 팁 → 나중에는 Pattern Inference의 출력이 여기로 흐른다.

### 10.2 "Earned Recognition" — 안전 장치에 추가할 원칙

7절의 안전 장치는 편향 강화 방지와 프라이버시를 다룬다. 여기에 하나를 추가해야 한다:

**보상/인정의 아부화(sycophancy) 방지.**

Overture의 핵심 가치는 anti-sycophancy인데, 보상 레이어가 그걸 훼손하면 모순이다. 원칙:

```
1. 침묵 > 범용 칭찬 — 잘한 게 없으면 아무 말도 하지 않는다
2. 과정을 칭찬, 사람을 칭찬하지 않는다 — "규제 리스크를 먼저 잡았다" (O) / "뛰어난 통찰력" (X)
3. 구체적이어야 한다 — 무엇을, 왜 잘했는지 명시
4. 희소해야 한다 — 매 단계마다 나오면 인플레이션
```

이건 70-20-10의 leverage(70%) 부분이 아부로 전락하지 않도록 하는 가드레일이다.

### 10.3 Counterfactual이 Cold Start 문제를 보완하는 방식

5절에서 데모를 "첫 번째 학습 데이터"로 활용하는 방안을 제시한다. Counterfactual 코칭은 이와 다른 각도에서 cold start를 보완한다:

```
데이터 없음  →  Counterfactual 팁 (인지적 씨앗)
                "만약 ~했다면 ~했을 것"
                → 다음 사용에서 의식적으로 시도 가능
                → 시도 자체가 행동 데이터가 됨

데모 완료    →  시드 프로필 (행동 데이터)
                가정 의심 성향, 역할 배분 성향 등

5+ 결정      →  패턴 기반 넛지 (통계적 근거)

15+ 결정     →  전체 적응형 루프 (Layer 2-4)
```

핵심 관찰: counterfactual 팁은 "미래의 데이터를 만드는 넛지"다. "만약 가정을 먼저 의심해봤다면"이라는 팁을 받은 사용자가 다음에 실제로 의심하면 — 그 행동이 데이터가 되고, 루프가 돌기 시작한다. Cold start 이전의 warm-up.

### 10.4 tone 분류와 70-20-10의 매핑

현재 구현된 tone 분류:

```
positive      →  leverage (70%)   — 잘한 것 인정
neutral       →  compensate (20%) — 보완 제안
counterfactual →  (cold start 전용, 비율 밖)
(없음)        →  challenge (10%)  — 의도적 도전 ← 아직 없다
```

`challenge` tone이 빠져 있다. 이건 Phase 5 (인지 개입)와 맞물리는데, 지금 당장은:

- Layer 4의 예시들: "보통 3회 반복하는데 이번엔 1회에 끝냈습니다. 의도적인 건가요?"
- 이런 메시지의 tone은 `positive`도 `neutral`도 아닌 **`challenge`**
- UI 색상: 골드(neutral) / 초록(positive) / 파랑(counterfactual) / **앰버 또는 오렌지(challenge)**

tone 분류에 `challenge`를 추가하면, 70-20-10 비율 모니터링의 UI 기반이 준비된다.

### 10.5 단일 반환 → 복수 반환: 구조적 한계

현재 `getStepCoaching()`은 단계당 **하나의 메시지만** 반환한다. positive가 나오면 neutral은 안 나오고, counterfactual이 나오면 거기서 끝난다.

적응형 루프에서는 같은 시점에 여러 신호가 경합한다:
- leverage: "가정 발견 수가 증가하고 있습니다" (positive)
- compensate: "이전 결정들에서 법무 관점이 빠졌습니다" (neutral)
- challenge: "이번에 첫 reframe을 바로 수락했는데, 의도적인가요?" (challenge)

이걸 지원하려면:
1. `getStepCoaching()` → `StepCoaching[]` 배열 반환
2. 70-20-10 비율에 따라 우선순위 정렬
3. UI에서 최대 2개까지 표시 (과하지 않게)
4. 세션 내에서 이미 보여준 톤을 추적하여 비율 조절

이건 Phase 2 구현 시 함께 진행하는 것이 자연스럽다.

### 10.6 프롬프트 주입(Application A)을 위한 연결점

이 문서의 가장 높은 가치 경로는 **A. 프롬프트 주입**이다. 현재 `context-chain.ts`에 `inject*Context()` 함수들이 있고, CLAUDE.md의 LLM Prompt Injection Guidelines가 이미 정의되어 있다:

> - Pattern data is reference only, not directive: "참고: ..." not "반드시 ..."
> - Content-based judgment is always primary, user patterns are secondary

이 가이드라인은 적응형 루프의 프롬프트 주입에 그대로 적용된다. 구현 시:

```typescript
// context-chain.ts에 추가될 함수 (예시)
function injectUserPatternContext(profile: ConcertmasterProfile): string {
  // "참고: 이 사용자는 최근 8회 결정 중 7회에서 첫 reframe을 수락했습니다."
  // "참고: 재무 리스크 관련 가정이 포함된 결정이 전체의 12%입니다."
  // → LLM이 이를 고려하되, 콘텐츠 판단이 우선
}
```

`concertmaster.ts`가 이미 프로필을 구축하므로, 이 함수의 입력은 준비되어 있다. 연결만 하면 된다.
