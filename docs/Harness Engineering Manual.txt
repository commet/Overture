# Harness Engineering Manual

## 1. 목적

이 문서는 특정 프로젝트의 개별 규칙 문서를 다시 열어보지 않아도, 다른 저장소에서 그대로 적용할 수 있는 공용 운영 매뉴얼이다.

대상:

- 사람 개발자

- Codex, Claude, Cursor 같은 코딩 에이전트

- 사람과 에이전트가 함께 작업하는 저장소

목표:

1. 에이전트가 같은 실수를 반복하지 않게 만든다.

2. 개발 규칙을 문서, 스킬, 훅, 체크리스트로 분리해 재사용 가능하게 만든다.

3. 새 프로젝트에 들어가도 빠르게 같은 운영 품질을 재현할 수 있게 만든다.

---

## 2. 핵심 개념

### 2.1 Harness란 무엇인가

Harness는 모델 자체가 아니라 모델 바깥에서 동작을 제어하는 운영 레이어다.

포함 요소:

- 시스템 지침: `AGENTS.md`, `CLAUDE.md`, 루트 운영 문서

- 상황별 지식: skills, playbooks, runbooks

- 도구 연결: MCP, CLI, lint/test/typecheck 명령

- 분리 실행: sub-agents, worker sessions

- 자동 검증: hooks, pre-commit checks, completion gates

- 기억과 피드백: docs, decisions, incident notes, failure review

핵심 원칙:

> 모델을 바꾸기 전에, 실패가 다시 발생하지 않도록 harness를 바꾼다.

### 2.2 좋은 harness의 조건

좋은 harness는 아래 조건을 만족해야 한다.

1. 짧다: 항상 필요한 규칙만 전역에 둔다.

2. 분리돼 있다: 조건별 규칙은 skill이나 별도 문서로 뺀다.

3. 강제력이 있다: 중요한 검증은 훅이나 체크리스트로 막는다.

4. 누적된다: 실패를 수정으로 끝내지 않고 시스템 규칙으로 흡수한다.

5. 이식 가능하다: 특정 저장소 배경을 몰라도 재사용할 수 있다.

---

## 3. 운영 레이어 설계

Harness는 한 파일에 모든 규칙을 밀어 넣지 않는다. 아래 4개 레이어로 나눈다.

### 3.1 Layer A: 헌법 문서

용도:

- 모든 작업에 항상 적용되는 불변 규칙

대표 위치:

- `AGENTS.md`

- `CLAUDE.md`

여기에 둘 것:

- 문서 우선 원칙

- 테스트/검증 필수 원칙

- 공통 모듈 재사용 원칙

- 파괴적 명령 금지

- 완료 보고 기준

여기에 두지 말 것:

- 긴 절차서

- 조건부 분기 규칙

- 특정 기능 전용 지침

- 상세 디렉터리 맵

### 3.2 Layer B: 상황별 실행 문서

용도:

- 특정 작업 유형에서만 필요한 규칙

예시:

- migration skill

- code review playbook

- incident recovery runbook

- API change checklist

형식:

- 문제 유형

- 언제 쓰는지

- 실행 순서

- 실패 시 대응

- 검증 방법

### 3.3 Layer C: 자동 강제 장치

용도:

- 사람이 잊거나 모델이 건너뛰기 쉬운 항목을 기계적으로 막는다

예시:

- pre-commit lint

- pre-push tests

- completion gate

- changed-files docs check

원칙:

- 사람 의지보다 자동화가 강하다

- 반복 위반이 발견되면 문장 규칙이 아니라 훅으로 승격한다

### 3.4 Layer D: 기억과 학습 문서

용도:

- 실패, 결정, 운영 사실을 미래 세션에서 재사용한다

예시:

- glossary

- architecture decision notes

- postmortem

- recurring failure ledger

원칙:

- 규칙은 추상화하고

- 사례는 기록으로 남긴다

---

## 4. 기본 운영 원칙

이 섹션은 어떤 프로젝트에도 그대로 옮길 수 있는 공용 규칙이다.

1. 모든 개발은 문서와 함께 진행한다.

2. 구조, 계약, 정책, API가 바뀌면 관련 문서를 즉시 갱신한다.

3. 기존 패턴, 공통 모듈, 공통 용어를 우선 사용한다.

4. 임시 코드, 중복 코드, 즉흥 규칙을 새 표준처럼 남기지 않는다.

5. 변경 전에는 영향 범위를 확인하고, 변경 후에는 실제 영향이 반영됐는지 검증한다.

6. 완료 주장은 검증 로그나 테스트 결과로 뒷받침한다.

7. 실패는 수동 수정으로 끝내지 않고, 재발 방지 규칙으로 흡수한다.

8. 새로운 개념이 생기면 먼저 이름과 의미를 정의한다.

9. 긴 전역 지시문보다 짧은 전역 지시문 + 필요한 순간의 세부 문서를 선호한다.

10. 에이전트는 추측보다 근거를 우선하고, 사람은 예외보다 시스템 개선을 우선한다.

---

## 5. 실패를 harness로 흡수하는 절차

이 문서의 핵심 운영 루프다.

### 5.1 실패가 발생했을 때

잘못된 대응:

- 지금만 고치고 끝낸다

- 모델이 원래 그렇다고 넘긴다

- 프롬프트를 길게 덕지덕지 붙인다

올바른 대응:

1. 실패 유형을 이름 붙인다.

2. 왜 발생했는지 분류한다.

3. 어느 레이어에서 막아야 하는지 정한다.

4. 규칙, skill, hook, 문서 중 하나로 흡수한다.

5. 다음 동일 작업에서 실제로 방지되는지 확인한다.

### 5.2 원인 분류

실패는 아래 중 하나로 분류한다.

1. 전역 규칙 부재

2. 특정 작업 절차 부재

3. 도구/검증 자동화 부재

4. 용어/문서 불일치

5. 과도한 컨텍스트 오염

6. 권한/책임 경계 불명확

### 5.3 흡수 대상 결정

아래 기준으로 넣는다.

- 모든 작업에 해당하면 `AGENTS.md`

- 특정 유형 작업에만 해당하면 skill/runbook

- 반복 위반을 강제로 막아야 하면 hook/script

- 미래 참고가 중요하면 docs/glossary/postmortem

---

## 6. 새 프로젝트에 도입하는 절차

### 6.1 Day 0 세팅

새 저장소에 들어가면 먼저 아래를 만든다.

1. 루트 `AGENTS.md`

2. `docs/` 아래 운영 문서 폴더

3. `docs/glossary/README.md`

4. 최소 1개 verification 명령

5. 최소 1개 completion checklist

### 6.2 Day 1 세팅

반복 작업이 보이면 아래를 만든다.

1. 첫 skill 1개

2. pre-commit 또는 completion hook 1개

3. failure log 문서 1개

### 6.3 Week 1 이후

매주 한 번 아래를 수행한다.

1. 실패 사례 회고

2. 반복 실패 패턴 추출

3. 규칙/skill/hook 승격 여부 결정

4. 오래된 규칙 삭제 또는 축약

---

## 7. 문서화 원칙

문서는 많을수록 좋은 것이 아니라, 경계가 명확할수록 좋다.

### 7.1 문서에 반드시 있어야 할 것

- 목적

- 적용 범위

- 강제 규칙

- 예외 처리

- 검증 방법

- 관련 문서

### 7.2 문서에 넣지 말아야 할 것

- 이미 코드가 더 정확히 보여주는 상세 구현

- 한번 쓰고 버릴 메모

- 같은 규칙의 중복 사본

- 조건이 과도하게 많은 의사결정 트리

### 7.3 용어 관리

새 용어가 생기면 아래를 동시에 정리한다.

1. 이름

2. 정의

3. 다른 용어와의 차이

4. 예시

5. 금지된 혼용 표현

---

## 8. Sub-agent 운영 원칙

Sub-agent는 역할놀이가 아니라 context firewall이다.

언제 쓰는가:

- 큰 검색 결과가 메인 스레드를 오염시킬 때

- 독립적인 조사/검증/리뷰 작업이 있을 때

- 구현과 검증을 병렬 처리할 수 있을 때

언제 쓰지 않는가:

- 지금 당장 결과가 필요한 막힌 핵심 작업

- 아주 작은 단일 파일 수정

- 읽지 않고 떠넘기는 탐색

원칙:

1. 메인 스레드는 방향과 통합을 맡는다.

2. 서브에이전트는 좁고 검증 가능한 일만 맡는다.

3. 서브에이전트 결과는 요약본만 상향 전달한다.

4. 같은 실수 패턴이 반복되면 sub-agent 추가보다 harness 개선을 먼저 본다.

---

## 9. Hooks와 검증 게이트

문장 규칙만으로는 부족하다. 아래 항목은 가능한 자동화한다.

필수 후보:

1. 완료 전 테스트 실행

2. 타입체크 실행

3. 린트 실행

4. 문서 변경 필요 여부 확인

5. 금지 명령 차단

6. 대량 변경 경고

우선순위:

1. 코드 품질 파괴를 막는 훅

2. 허위 완료 보고를 막는 훅

3. 문서 불일치를 막는 훅

---

## 10. 유지보수 규칙

Harness는 계속 커지면 오히려 성능을 떨어뜨린다. 유지보수가 필요하다.

### 10.1 정기 정리 기준

아래 규칙은 주기적으로 삭제 또는 축약 검토한다.

- 최근 8주 이상 한 번도 필요 없던 규칙

- 같은 내용의 중복 규칙

- 원래 문제를 더 이상 일으키지 않는 임시 규칙

- 전역에 있을 이유가 없는 상세 규칙

### 10.2 성능 경고 신호

아래 현상이 보이면 harness가 비대해진 것이다.

- 에이전트가 규칙을 자주 충돌시키며 해석한다

- 전역 문서가 지나치게 길다

- 같은 규칙이 여러 문서에 복제돼 있다

- tool 선택이나 절차 선택에 시간을 낭비한다

- 완료 전에 꼭 필요한 검증을 여전히 자주 빠뜨린다

대응:

1. 전역 규칙을 줄인다.

2. 조건부 규칙을 skills로 분리한다.

3. 반복 체크를 hooks로 옮긴다.

4. 오래된 문서를 archive로 내린다.

---

## 11. 금지 패턴

아래는 공통적으로 피해야 할 anti-pattern이다.

1. 실패할 때마다 전역 프롬프트만 길게 늘리는 것

2. 사람이 늘 기억할 거라고 가정하는 것

3. 에이전트가 디렉터리 구조를 모른다고 보고 문서에 트리 전체를 붙이는 것

4. 작은 문제를 새 추상화/새 의존성으로 덮는 것

5. 문서와 실제 운영 계약을 분리해 두는 것

6. 서브에이전트를 역할 장식처럼 남용하는 것

7. 검증 없이 “완료”라고 보고하는 것

---

## 12. 빠른 도입 체크리스트

새 프로젝트에서 이 매뉴얼을 도입할 때 최소 순서는 아래다.

1. 루트 `AGENTS.md` 작성

2. 검증 명령 3개 정의: lint, typecheck, test

3. glossary 문서 생성

4. 반복 작업 1개를 skill로 분리

5. completion hook 1개 추가

6. 실패 회고 문서 시작

이 6개만 있어도 기본 harness는 작동한다.

---

## 부록 A. 최소형 `AGENTS.md` 템플릿

```md

# Project Operating Contract

## Core Rules

- Solve the task directly when safe.

- Update related docs when contracts, APIs, or architecture change.

- Reuse existing modules and patterns before adding new abstractions.

- Verify before claiming completion.

- Do not use destructive commands unless explicitly approved.

## Workflow

- Read relevant docs before editing.

- Keep diffs small and reversible.

- Prefer project-standard commands for lint, typecheck, and tests.

- If a mistake repeats, encode the prevention into the harness.

## Completion

- Report changed files.

- Report verification actually run.

- Report remaining risks or gaps.

```

사용 규칙:

- 40~60줄 안쪽 유지

- 모든 작업에 적용되는 내용만 둔다

- 기능별 규칙은 넣지 않는다

---

## 부록 B. Skill 템플릿

```md

---

name: api-endpoint-workflow

description: Use when adding or changing API endpoints.

---

## When To Use

- New endpoint

- Request/response schema change

- Auth/validation change

## Steps

1. Read the API contract docs.

2. Check existing endpoint patterns.

3. Update schema first.

4. Implement handler.

5. Add or update tests.

6. Update docs and examples.

## Verification

- Run endpoint tests

- Run typecheck

- Confirm docs changed if contract changed

## Common Mistakes

- Forgetting schema updates

- Changing response shape without docs

- Skipping auth/error-path tests

```

분리 기준:

- 특정 작업에서만 필요하면 skill로 뺀다

- 1개 skill은 1개 작업 유형만 다룬다

---

## 부록 C. Hook/Checklist 템플릿

### C.1 Completion Gate Checklist

```md

Before declaring completion:

1. Did I run the relevant tests?

2. Did I run lint/typecheck if the change affects typed or shared code?

3. Did I update related docs/contracts?

4. Did I verify the user-visible behavior?

5. Did I record remaining risks honestly?

```

### C.2 반복 위반 승격 규칙

아래 조건 중 하나면 hook 후보로 올린다.

1. 같은 누락이 2회 이상 발생

2. 사람이 자주 까먹는 검증

3. 실패 비용이 큰 항목

4. 완료 보고 품질을 직접 흔드는 항목

---

## 부록 E. 도입 순서 템플릿

아래 순서로 적용하면 된다.

1. 이 문서를 프로젝트 루트 운영 문서와 함께 읽는다.

2. 최소형 `AGENTS.md`를 프로젝트 실정에 맞게 채운다.

3. 검증 명령 3개를 확정한다.

4. glossary를 만든다.

5. 첫 반복 작업을 skill로 분리한다.

6. completion checklist를 hook 또는 운영 규칙으로 강제한다.

7. 첫 주 실패 사례부터 weekly review를 시작한다.

## 13. 한 줄 요약

좋은 harness는 "에이전트가 잘하길 기대하는 문장 모음"이 아니라, 실패를 규칙, 문서, skill, hook으로 누적 흡수하는 운영 시스템이다.
