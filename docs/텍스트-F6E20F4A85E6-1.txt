AI 에이전트의 "스킬"이 
슬래시 명령어 수준에 머물러 있다는 지적이 나왔습니다.

Random Labs가 Slate 블로그에서 던진 질문이에요.

에이전트 스킬은 사람의 스킬과 같아야 하는데,
지금은 사용자가 직접 발동하는 슬래시 명령어에 불과하다는 겁니다.

스킬을 행동으로 바꾸면 어떻게 되는지, 
그리고 그걸 엮으면 뭐가 가능한지 나옵니다.

Skill Chaining, 
에이전트 스킬이 슬래시 명령어를 벗어나는 법입니다.￼

Slate는 Random Labs가 만드는 AI 코딩 에이전트입니다.

Claude Code나 Cursor 같은 도구를 떠올리면 돼요.

터미널에서 코드를 읽고, 고치고, 테스트하는 에이전트인데,
내부적으로 "스레드"라는 격리된 작업 단위로 움직입니다.

이 팀이 자기네 블로그에서 에이전트 스킬의 구조적 문제를 짚었어요.

스킬이 왜 슬래시 명령어에 머물러 있는지, 
어떻게 바꿨야 하는지.￼

지금 에이전트 스킬이 어떻게 작동하는지 보면요.

`/frontend-design` 같은 슬래시 명령어를 사용자가 직접 입력합니다.

스킬 텍스트가 컨텍스트 창 상단에 고정돼요.

작업이 끝나면 빠집니다.￼

문제는 두 가지입니다.

첫째, 
사용자가 언제 어떤 스킬을 쓸지 직접 판단해야 합니다.

유튜브로 타이어 교체법을 배운 사람이
매번 영상을 틀어놓고 따라 하는 것과 비슷하죠.

한 번 배웠으면 상황에 맞춰 알아서 꺼내 쓸 수 있어야 합니다.

둘째, 
규칙(rules)과 스킬의 경계가 흐릿합니다.

규칙은 수만 토큰을 컨텍스트에 상시 주입해요.

스킬은 필요할 때만 넣고 빼는 구조인데,
결국 사용자가 수동으로 넣고 빼고 있으니 차이가 희미합니다.￼

Slate는 스킬을 "행동"으로 재정의했습니다.

읽는 것이 아니라 수행하는 것.
상황에 맞춰 에이전트가 스스로 꺼내 쓰는 맥락적 행동이라는 거예요.

이걸 가능하게 만드는 개념이 하나 있습니다.
"knowledge overhang"이라고 불러요.

모델이 할 줄 아는 것과 
실제로 하기로 선택하는 것 사이에 간극이 있습니다.

스킬은 그 간극을 메우는 도구예요.

모델이 이미 알고 있지만 스스로 꺼내지 않는 행동을,
맥락 주입으로 끌어내는 겁니다.￼

여기서 문제가 하나 걸립니다. 사용자와 대화가 필요한 스킬이에요.
코드를 짜는 스킬은 백그라운드에서 혼자 돌면 됩니다.

그런데 설계를 잡는 스킬은 사용자한테 물어봐야 해요.
"이 구조로 갈까요?" 같은 확인이 필요하죠.

Slate는 세 가지 방법을 시도했습니다.

팝업 다이얼로그. 서브에이전트가 질문을 띤우는 방식인데,
한 번에 한 질문밖에 못 해서 대화가 안 됩니다.

스레드 에스컬레이션. 
서브에이전트가 사용자 확인이 필요하면

메인 에이전트로 올려보내는 방식인데,
누가 주도권을 갖는지 꼬이기 시작합니다.

세 번째가 포킹입니다.

포킹은 에이전트를 복제해서 대화를 이어가는 겁니다.

메인 에이전트와 동일한 사본을 만들어서
사용자와 직접 대화하게 합니다.

대화가 끝나면 결과만 돌려주고 사라져요.

사용자 입장에서는 그냥 같은 에이전트와 계속 대화하는 느낌이에요.
뒤에서는 격리된 에피소드가 돌아가는 겁니다.

이렇게 되면 스킬이 두 갈래로 나늉니다.

혼자 돌아가는 액션 스킬은 스레드로.
사용자와 대화가 필요한 인터랙티브 스킬은 포크로.

에이전트가 상황에 따라 어느 쪽을 쓸지 결정합니다.￼

진짜 재밌는 건 여기서부터입니다. 스킬이 스킬을 부립니다.

오케스트레이션 스킬이라는 개념이에요.

스크립트나 도구를 참조하는 게 아니라, 다른 스킬을 참조하는 스킬입니다.

예를 들면 이렇습니다.

"기능 구현" 오케스트레이션 스킬이 활성화되면,
먼저 `plan` 스킬을 포크로 실행합니다. 
사용자와 설계를 잡아요.

설계가 끝나면 `implement` 스킬을 스레드로 돌립니다. 
백그라운드에서 코딩.

코딩이 끝나면 `review` 스킬을 포크로 실행합니다. 
사용자와 함께 검토.

검토 통과하면 `qa` 스킬을 포크로 실행합니다.￼

plan → implement → review → QA.

앞 단계의 출력이 다음 단계의 입력이 됩니다.

세션 끝에 `/review` 치는 걸 잊어버리는 문제가 사라져요.

오케스트레이션 스킬이 순서를 알고 있으니까요.￼

이 구조가 OS랑 닮았다는 반응이 나옵니다.

스레드, 포킹, 권한 모델, 컨텍스트를 프로세스 메모리처럼 다루는 방식.
운영체제가 프로세스를 관리하는 패턴과 겹칩니다.

우연이 아닐 수 있어요.

에이전트가 복잡해질수록 필요한 구조가
결국 OS가 이미 풀어놓은 문제와 같아지는 거니까요.

포킹은 2026년 4월 1일 기준 알파 단계입니다.

아직 프로덕션은 아니에요.￼

방향을 되짚어봅니다.

지금까지 에이전트 스킬은 
사용자가 고르고, 사용자가 발동하고, 사용자가 순서를 챙겼습니다.

오케스트레이션 스킬이 작동하면, 
이 세 가지가 전부 에이전트로 넘어갑니다.

스킬이 명령어에서 행동으로 바뀌는 겁니다.

사용자가 시키는 도구에서, 에이전트가 판단하는 역량으로.￼
