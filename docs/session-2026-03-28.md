## 2. Judgment Vitality Engine

> "서로를 지탱함을 통해서 얻은 안정감과 체계화 때문에 이들은 경직되어 간다."
> — 컴퓨터와 마음 기말고사 답안 (2014, 서울대 김재인 교수님 수업)

**핵심 질문: "이 판단 과정은 살아있는가, 죽어있는가?"**

2014년 서울대 "컴퓨터와 마음" 수업에서 Bateson의 *Mind and Nature*을 원서로 읽으며 보낸 한 학기. 12년 후 Overture를 만들다 막혔을 때, 당시 기말 답안의 이 문장이 — AI 아첨, 형식적 수렴, 프로세스 게이밍 등 모든 문제의 본질이었다.

User↔AI가 서로 동의하면 경직된다. User↔Persona가 예측 가능해지면 경직된다. User↔System을 게이밍하면 경직된다. System↔System이 DQ 인플레이션을 일으키면 경직된다. 10개의 이론적 기반을 10개의 개별 기능이 아닌, **하나의 엔진**으로 통합했다.

### 아키텍처

concertmaster.ts 패턴을 따르는 **읽기 전용 레이어**. 기존 store/signal에서 읽고, recordSignal()로 쓰고, concertmaster coaching으로 전달.

```
각 단계 eval 기록 시 → buildStageFingerprint → stage_fingerprint signal
                                                        ↓
리허설 완료 → translateApprovalsToPlan → translated_approvals 저장
                                                        ↓
DQ 계산 시 → assessVitality → VitalityAssessment 저장 + signal
                                                        ↓
Concertmaster 호출 시 → vitality insights/coaching → 사용자에게 표시
```

### Phase 0: Actor 모델 전환

기존 `ai | human | both` → `human | ai | human→ai | ai→human`

| 값 | 의미 | 예시 |
|---|------|------|
| `human` | 사람만 | 이해관계자 미팅, 최종 의사결정 |
| `ai` | AI만 | 데이터 수집, 분석 자동화 |
| `human→ai` | 사람이 판단/방향, AI가 실행 | "전략 방향 잡으면 AI가 초안" |
| `ai→human` | AI가 생성/분석, 사람이 결정 | "AI가 옵션 제시하면 내가 선택" |

*"경계가 아니라 관계를 설계하라"* (2014 기말 3번 답안) — 누가 하느냐가 아니라 어떤 순서로 협업하느냐.

### Phase 1: Instrumentation — 볼 수 있게 만들기

| 함수 | 역할 |
|------|------|
| `buildStageFingerprint(phase, item)` | 각 단계 출력의 구조적 스냅샷 |
| `translateApprovalsToPlan(record, steps, personas)` | 페르소나 승인조건 → step 레벨 매핑 |
| `buildProvenanceChain(reframe, recast, feedback, refine)` | 인사이트 출처 추적 — 어떤 가정이 어디까지 영향을 미쳤는가 |

### Phase 2: Measurement — γ 계산

| 함수 | 역할 |
|------|------|
| `computeStageGamma(fingerprint)` | 각 단계가 생산한 진짜 부가가치 (0-1) |
| `computeProjectGamma(reframe, recast, feedback, refine)` | 전체 프로젝트 γ + provenance bonus |
| `measureFeedbackNovelty(initial, reReview)` | 피드백 신선도 (0=동일, 1=완전히 다름) |

**각 단계별 γ 측정 기준:**
- **Reframe**: 질문이 바뀌었는가, 가정의 수와 축 다양성
- **Recast**: actor 다양성, 체크포인트 비율, 크리티컬 패스
- **Rehearse**: 리스크 다양성, unspoken 리스크 비율, 승인조건 수
- **Refine**: 이슈 해결량, 조건 충족률, iteration 횟수

빈 프로젝트 = γ 0.5 (neutral). 불완전 프로젝트 ≠ dead.

### Phase 3: Detection — 경직 감지

7종 경직 신호:

| 카테고리 | 신호 | severity |
|---------|------|----------|
| User↔AI | `convergence_too_fast` — 1회 수렴 + low γ | 0.3 |
| User↔AI | `frame_unchanged` — surface ≈ reframed (유사도 >0.8) | 0.6 |
| User↔Persona | `feedback_predictable` — 리뷰 vs 리리뷰 novelty <0.2 | 0.4 |
| User↔Persona | `same_persona_set` — 3회+ 연속 같은 세트 | 0.3 |
| User↔System | `low_gamma_high_dq` — γ <0.2 but DQ >70 | 0.7 |
| User↔System | `translated_approval_ignored` — 구체적 제안 미반영 | 0.5 |
| User↔System | `actor_pattern_rigid` — 같은 actor 패턴 반복 | 0.3 |

### 핵심 공식

**vitality = γ × (1 - rigidity_score)**

| 범위 | 분류 | 의미 |
|------|------|------|
| >0.7 | alive | 진짜 판단 진행중 |
| 0.4-0.7 | coasting | 루틴화 시작 |
| 0.2-0.4 | performing | 형식적 수행 |
| <0.2 | dead | 순수 compliance |

### Phase 4: Response — 티어별 개입

| Tier | 조건 | 대응 |
|------|------|------|
| 1 | 첫 발생 | 힌트 (neutral) |
| 2 | 2회차 | 명시적 플래그 |
| 3 | 3회+ | 구조적 개입 제안 |
| 4 | 5회+ 행동변화 없음 | 메타코칭: *"서로 지탱하면 경직된다 — 이 AI와의 관계도 마찬가지일 수 있습니다."* |

### Devil's Advocate 리뷰 → 8건 즉시 수정

| 수정 | 내용 |
|------|------|
| computeGamma 재설계 | 크로스 페이즈 비교 → 각 페이즈 자체 품질 측정 (`computeStageGamma`) |
| 임계값 0.15 → 0.3 | translateApprovalsToPlan 오탐 방지 |
| 불완전 프로젝트 보호 | 빈 fingerprint = 0.5 (neutral), not 0 (dead) |
| vitality 공식 일관성 | maxRigidity → avg rigidityScore로 통일 |
| fingerprint 중복 제거 | computeProjectGamma가 반환한 fingerprints 재사용 |
| 바 차트 세그먼트 추가 | human→ai, ai→human 시각화 |
| LLM 프롬프트 업데이트 | Recast AI에 새 actor 타입 안내 |
| ActorToggle 폴백 | 레거시 'both' → 'human→ai' 하이라이트 |

- 단위 테스트: 40개 통과

### 알려진 한계

- **γ는 구조적 프록시.** "사고가 진짜 변했는가"가 아니라 "구조적 풍부함" 측정. outcome 데이터 축적 후 calibrate.
- **bag-of-words cosine similarity.** 한국어 합성어 문제 있음. Transformers.js (multilingual MiniLM) 교체 검토 중.
- **coaching 효과 미검증.** 경직 경고가 행동 변화를 일으키는지는 outcome tracking 성숙 후 검증.

### 설계 원칙

**1 eval = 1 prompt** (single-aspect judging, Zheng et al.)

각 eval에 4개 필드:
- `question`: 무엇을 판단하나
- `decision_line`: **어떻게** 판단하나 (judge의 사고 도구)
- `pass_example`: 이 수준이면 통과
- `fail_example`: 이 수준이면 탈락

### LLM Judge 2가지 모드

- `buildContentJudgePrompts()`: 1 eval = 1 prompt (정확도 우선)
- `buildBatchJudgePrompt()`: 전체 1 prompt (비용 우선)

### 풀 파이프라인 QA — "AI 코드 리뷰 어시스턴트"

4개 .overture/ 파일을 SKILL.md 사양 대비 대조.

**가장 인상적인 체이닝:**
```
reframe: "자동화의 경쟁자는 멘토링 욕구"
  → rehearse: "시니어에게 리뷰는 영향력 행사"
  → refine: thesis 피벗 "시간 절약 → 영향력 확장"
```

### 보류한 것: near-miss exemplar

- 경계선 예시 추가 시 judge 기준이 흐려지는 위험
- 5가지 패턴 식별 ("하위 질문 교체", "도메인 포장", "깊이 1단계 부족", "가짜 구체성", "교과서 정답")
- **결론: decision_line을 더 날카롭게 다듬는 게 ROI 높음**

### 기술적 보완 가능성 (embedding)

| 방법 | 용도 | LLM 대체 여부 |
|------|------|-------------|
| 원본↔재정의 유사도 | frame_shift 자동 측정 | 완전 대체 |
| generic assumption bank | 뻔한 가정 사전 비교 | 완전 대체 |

→ embedding으로 수치화 가능한 기준은 LLM judge에서 빼고, judge는 진짜 판단이 필요한 것만 맡기기.

## 4. Pipeline Overhaul

SKILL.md 5개 전면 개편. 3개 테스트 파이프라인 실행으로 검증 (뉴스레터 수익화, AI 코드 리뷰 SaaS, 프리랜서 정산 도구).

### 에이전트급 UX

- **Quick actions** (0/1/2/3) + numbered edit targets (a/b/c/d) — 1번 입력으로 의도 전달
- **양방향 탐색** (← 0) — 이전 단계로 돌아가기 + 6가지 engine-driven backward 규칙:
  - "전원 거부 → /recast로"
  - "thesis 공격 → /recast로"
  - "문제 존재 의심 → /reframe으로"
- **Reflection blocks** — 처리 중 "thinking appetizer" 표시
- **Severity-based /refine nudge** after /rehearse

### 카드 재설계

- **76자 폭 통일** (이전 46-55 불일치)
- 1-line headers + ━━━
- Pipe tables with separators
- Card blocks (┌│└)
- Left-pipe only for assumptions (한국어 깨짐 방지)

### 저널 풍부화

- 가정/리스크/침묵의 **실제 내용** 기록 (숫자만이 아니라)
- Critique resolution tracking (rehearse → refine 연결)
- Pipeline progress indicator: `reframe ✓ → recast ✓ → rehearse · refine ·`
- Unique headers (date + skill + topic slug)

### 패턴 적응 (저널 기반)

- Blind spot forcing — 같은 영역 2회+ 시 강제 질문
- Confidence warnings — 과신 패턴 감지
- Strategy repetition alerts — 같은 전략 3회+ 반복 시 경고
- Cross-run topic linking, assumption follow-up
- Per-skill readiness score (██░░░ bar)

### Context 지원 현황

```
/reframe  → context: build | decide  ← 인터뷰 질문, 가정 축, 전략이 분기
/recast   → context 상속              ← governing_idea vs product thesis
/rehearse → context 상속              ← 페르소나 선택에 반영
/refine   → context 상속              ← 수렴 기준 동일
/overture → context 자동 감지         ← 딜리버블이 분기
```

## 5. 자기 비평 + 개선 로드맵

`.overture/critique-and-roadmap.md` — 현재 상태의 가혹한 평가 + 구조적 개선 방향.

### 진짜 잘 된 것

1. **Context Contract chain** — reframe → recast → rehearse → refine 사이에 구조화된 데이터가 흘러감
2. **Build/Decide 분기** — 질문 체계, 가정 차원, 산출물 형태가 컨텍스트에 따라 전부 바뀜
3. **Engine-driven backward recommendation** — 잘못된 방향으로 전진하는 걸 실제로 막음
4. **저널의 실제 출력 품질** — thesis가 "시간 절약" → "시니어 영향력 확장"으로 전환. 생각이 발전하는 과정이 보임

### 치명적 결함 5개

| # | 결함 | 비유 |
|---|------|------|
| 1 | **결과 피드백 부재** — DQ 72점이 좋은 결정이었는지 영원히 모름 | 테스트 없는 CI/CD — 빌드 성공만 알고 작동 여부는 모름 |
| 2 | **Decision Theater** — 페르소나 비판 + DQ = 인지적 위안. 실제 검증 0건으로도 "충분히 검증했다" 착각 | 잘 구조화된 추측에 숫자를 붙인 것 |
| 3 | **LLM 자기 대화의 천장** — 같은 LLM이 역할극. "그럴듯한 비판"은 만들지만 "예상 못 한 비판"에 구조적 한계 | 진짜 stakeholder가 주는 비대칭 정보, 예측 불가능한 연결, skin in the game이 없음 |
| 4 | **삼중 편향** — 사용자가 확신도 자기 평가 → 이걸 기반으로 전략 선택 → LLM이 결과 자기 채점. 외부 신호 제로 | Dunning-Kruger → 오염된 입력 → 오염된 출력 |
| 5 | **Adaptive rules 죽은 코드** — 핵심 기능이 3회+ 사용 필요, 대부분 1-3번으로 판단 | 가장 강력한 차별 기능이 발동조차 안 함 |

### 부차적 결함 4개

1. Blind spot이 "그럴듯한 관찰" 수준 (PM 101, 진짜 불편한 진실이 아님)
2. Build context가 주말 프로젝트에 과함 (analysis paralysis를 analysis로 치료)
3. 포맷이 Claude Code 터미널에 종속 (Slack/Notion에서 깨짐)
4. DQ 점수의 자기 채점 (학생이 자기 시험 채점)

### 근본 문제

> LLM이 만들고, LLM이 평가하고, 사용자가 자기 보고하는 closed system에서 외부 신호가 없다.

풀 방향 3가지:
1. **외부 신호를 주입한다** (evidence grounding)
2. **사용자 자신이 모르는 것을 드러낸다** (metacognition)
3. **명시적 피드백 없이 결과를 추론한다** (implicit signals)

### 한 문장 판단

> "혼자 일하는 사람에게 '똑똑한 동료 한 명이 30분 같이 생각해주는 것'의 80%를 제공한다. 개선안의 목표는 80%를 95%로 올리는 것이 아니라, 80%의 한계를 정직하게 인정하면서 사용자가 나머지 20%를 직접 채우도록 유도하는 것이다."

### 개선 설계 — Crux가 전체 파이프라인을 관통

**핵심 설계 원칙:**
- Crux(이것만 틀리면 전부 무너지는 가정 1개)가 /reframe에서 식별되어 이후 모든 단계의 초점이 됨
- Evidence는 /rehearse가 아니라 /recast에서 수집 — spec 설계 전에 현실 주입
- Anti-theater는 면책 조항이 아니라 행동 목록
- Conviction Arc는 /overture 전용 — 시작과 끝 2번만, 가치는 delta에

### 데이터 흐름

```
/reframe
  ├─ R1. Absence Analysis → 빈 영역을 가정 생성 엔진의 입력으로
  ├─ R2. Crux Identification → Contract.crux (전 단계 전파)
  ├─ R3. Epistemic Status (단계별 근거 수준 2줄)
  └─ R4. MVT 제안 (crux 기반, 1회)
       ↓ Contract: crux, assumptions, evidence_basis

/recast
  ├─ C1. Evidence Gathering (WebSearch) → Contract.evidence
  ├─ C2. Outcome Hook → journal.outcome_check + check_after 날짜
  └─ spec 설계에 evidence 반영
       ↓ Contract: evidence, crux (상속), spec, personas

/rehearse
  ├─ H1. Evidence-grounded personas (evidence 상속 + failure case 추가)
  └─ H2. Crux 집중 공격 (모든 페르소나의 첫 공격 대상 = crux)
       ↓ Contract: risks, critiques, crux_attack_result

/refine + /overture
  ├─ F1. Validation Checklist (crux 기반 행동 목록, "다음?" 대신)
  ├─ F2. Conviction Delta (/overture만, 시작/끝 2번)
  └─ F3. Epistemic Status (전체 파이프라인 누적)

Cross-cutting:
  └─ Passive Outcome Trigger (모든 스킬 Before starting)
      → /patterns의 calibration 데이터
```

### 해자가 될 수 있는 것 vs 되기 어려운 것

**될 수 있는 것:**
1. Crux 관통 설계 — 프롬프트 복사만으로 데이터 흐름 복제 어려움
2. Evidence grounding — 실시간 데이터 주입, 프롬프트만으로 복제 불가
3. Outcome-linked calibration — 시간이 갈수록 강해지는 유일한 장벽
4. Epistemic honesty — "이 분석의 70%는 LLM 추론입니다" 선언하는 도구가 시장에 없음
5. Metacognitive tools — 사용자의 사고 자체를 분석 대상으로 만듦

**되기 어려운 것:**
- 4R 파이프라인 구조 (프롬프트 복제 가능)
- 페르소나 역할극 (아무 LLM이나)

## 6. 사고 변화 측정 리서치

γ가 구조적 프록시만 측정하고 "진짜 사고 변화"를 못 잡는 문제. 데이터 축적 후 Phase 1.5로 구현 예정.

### 즉시 구현 가능 (의존성 0)

**행동 신호:**
- 삭제 비율 — 삭제 키스트로크 / 전체 (높으면 = 재작성 = 진짜 사고)
- 편집 위치 분산 — 중간을 고치면 구조 변경, 끝에만 추가하면 순응
- 뒤로 돌아감 — 리허설 후 리프레임으로 수정 = 강력한 사고 변화 신호
- 체류 시간 — 특정 단계에서 오래 머무름 = 숙고
- 출처: EDM 2021, LAK 2025

**담화 표지 감지:**
- "하지만", "사실은", "다시 생각해보면", "오히려", "결국" 등 전환 표지를 정규식으로 카운트
- 의존성 0, 한국어/영어 모두 가능

**IDF 가중 용어 변화:**
- 현재 bag-of-words cosine에 IDF 가중치 추가
- 원문에만 있는 용어 + 수정본에만 있는 용어의 비율 = 사고 변화 지수

### 중기 구현 (Transformers.js 도입 시)

- `paraphrase-multilingual-MiniLM-L12-v2` (50MB, ONNX, 한국어 지원)
- 브라우저에서 100-200ms/문장, 캐싱 가능
- 문장 단위 정보 이득: (새로운 + 삭제된) / 전체 = thinking change index
- 출처: NAACL 2024 semantic drift

### 참고 논문

- Fokkinga et al. (PMC 2023) — pre/post 신념 측정
- CHI 2025 — AI 신뢰도 높을수록 비판적 사고 감소
- Mandel 2024 — ACH는 판단 품질 향상에 효과 미미 (프로세스 준수 ≠ 사고 품질)
- DQI (Discourse Quality Index) — 숙의 민주주의 품질 측정 5차원
- AQuA 2024 — 통합 숙의 품질 점수

## 7. 버그 수정

### DQ 캐시 무효화 (HIGH)

`computeDecisionQuality`가 projectId만으로 캐시, 무효화 메커니즘 없음. 파이프라인 진행 후 재호출해도 옛 점수 반환.

**수정:** `DQInput`에 `force?: boolean` 추가. `force: true` 시 캐시 무시 + 기존 엔트리 replace. `RefineStep.tsx`에서 `force: true`로 호출.

### safeCompare bypass (CRITICAL)

`b = a` 재할당 후 `a.length !== b.length`가 항상 false → 길이가 다른 문자열도 `true` 반환.

**영향:** cron 엔드포인트의 CRON_SECRET 인증이 길이가 다른 토큰에 대해 bypass 가능.
**수정:** `lengthMismatch` 플래그를 재할당 전에 캡처. 비교 대상을 `compareTarget`으로 분리.

### concertmaster crash (MEDIUM)

`concertmaster.ts:338` — `latest.signals.length`에서 signals undefined일 때 crash.
**수정:** `latest.signals?.length` optional chaining.
