# Overture Compact Pivot — 이정표 문서

**날짜:** 2026-03-30
**트리거:** 종화 (스타트업 운영자) 피드백 세션
**결과:** Plugin 5개 스킬 전면 재작성 + 랜딩 페이지 4개 컴포넌트 리라이트

---

## 1. 종화 피드백 원문 (핵심 발췌)

### 피드백 요약

종화는 스타트업을 운영하는 친구로, Overture에 대해 다음 피드백을 줬다:

1. **"잘 하던 걸 잘 하는 건 임팩트 없다."**
   - 개발자가 기획안을 쓰는 건 아예 안 해본 것.
   - 기존에 잘 하던 걸 잘 하게 해주는 건 임팩트가 약하다.
   - 진짜 임팩트는 **자기 영역 밖으로 던져진 사람을 살려주는 것**.

2. **"사람은 배우고 싶어하지 않는다."**
   - "너를 잘 할 수 있게 도와주겠다" → 사람들이 안 한다.
   - "내가 다 해봤으니까 따라오면 돼" → 이게 먹힌다.
   - 교육 프레이밍이 아니라 권위 프레이밍.

3. **"Use case로 전달하기."**
   - 추상적 프레임워크가 아니라, "전략일을 해야 하는데 나도 적용할 수 있겠다"는 구체적 상황.

4. **"제품이 너무 많은 문제를 해결하려 한다."**
   - "기획을 해야 하는 개발자를 위한 양성 프로그램"으로 좁히기.
   - Phase 1: 기획하라고 된 개발자 → Phase 2: 금융 등 산업 확장.

5. **"시리즈 콘텐츠로 케이스 쌓기."**
   - 헤드라인: "개발자로 입사한 나에게 대표님이 기획을 해보라고 했다"
   - Pain → Tool → Outcome (연봉, 인정)

## 2. 이후 추가 논의에서 나온 방향

종화 피드백 이후 추가로 정리된 구체적 방향:

### 배포/마케팅 스토리
- 어그로 헤드라인으로 시작: "기획하셔야 되는 개발자가 있냐"
- 백엔드 엔지니어인데 대표님이 2주 안에 기획서 써오라고 한 상황
- "말로 조언해주다가 아예 도구로 만들었다. 이제 나한테 물어보지 말고 Overture에게 물어보시라."
- SK 전략기획 5년차 + 수능 만점의 노하우를 녹인 도구 (마케팅 글에서 사용, 제품 UI에서는 은근하게)

### 제품 UX 핵심
- **호흡이 너무 길다** — 4단계를 거쳐야 결과물이 나오는 구조를 깨야 함
- **바로바로 1차 결과** — 입력 하나 → 결과 하나 → 입력 하나 → 결과 하나
- **페르소나/리허설은 필수가 아님** — "대표님은 뭐라고 할까?"가 뜨면 궁금해서 눌러보게 되는 것
- **짧은 호흡으로 가다가 마지막에 전체 조합도 볼 수 있게**

### 사용자 인터뷰 전략
- 좋은 후기 남긴 사람과 직접 만나기 (전화/커피챗)
- 5명은 직접 옆에서 도와주면서 사용 양상 관찰
- use case를 실제로 확보 → 홍보 소재
- "대표님이 'Overture 써봐' 할 수 있는" 플로우 만들기

### 톤 원칙
- "너무 고귀하거나 세련되게 가면 안 됨"
- 피부에 와닿는 이야기와 use case
- 편하게 짧은 호흡으로 써보게

## 3. 구현 전 핵심 의사결정

### 메타포 수준: "은은하게 유지"
- Reframe/Recast/Rehearse/Refine 영어 이름 유지 (스킬 명령어)
- 한국어 레이블: "악보 해석/편곡/리허설/합주" → "문제 재정의/실행 설계/사전 검증/수정 반영"
- "Overture" 제품명 유지
- UI 장식 요소(MusicalElements)는 건드리지 않음

### AskUserQuestion 도구 활용
- 모든 스킬의 `allowed-tools`에 이미 AskUserQuestion 선언되어 있었음
- 그런데 실제로는 코드블록+숫자 입력 패턴을 사용 중이었음
- **변경:** SKILL.md 인스트럭션에서 "AskUserQuestion 도구를 사용해서 질문하라"고 명시
- 결과: 터미널 하단 화살표 선택 UI → 숫자 타이핑보다 직관적

### 인터뷰 유지 (제거하지 않음)
- 초기 계획: 인터뷰 없애거나 1개로 축소
- **수정:** 인터뷰는 내용 파악에 필수. 다만 **인터뷰 전에 1차 결과를 먼저 보여줌**
- 인터뷰는 "결과를 생성하기 위한 것"이 아니라 "결과를 업데이트하기 위한 것"

### 판단자 동적 결정
- 초기: "대표님은 뭐라고 할까?"로 고정
- **수정:** 인터뷰 Q1 "이 결과물을 누가 최종 판단해?"에서 파악
  - 대표님 → "대표님은 뭐라고 할까?"
  - 팀장 → "팀장님은 뭐라고 할까?"
  - 투자자 → "투자자는 뭐라고 할까?"
  - 모름 → "가장 까다로운 사람은 뭐라고 할까?"

### Hero 카피 — 은근한 신뢰성
- ❌ "SK 전략기획 5년차가 밤새며 정립한 방법론" (제품 내 전면)
- ✅ "인지과학 + 전략기획 실무 기반 프레임워크" (작은 텍스트)
- 5년차/수능 스토리는 마케팅 글(블로그, SNS)에서 풀기

## 4. 구현 내역

### Phase 1: Plugin 스킬 5개 재작성

#### `/overture` SKILL.md — 완전 재작성
- **Before:** 425줄, 4단계 순차 파이프라인 (reframe→recast→rehearse→refine), 각 단계 완료 후 다음
- **After:** ~280줄, Progressive flow
- **새 Flow:**
  ```
  입력 → 즉시 1차 결과 (30초, 인터뷰 전)
       → AskUserQuestion 인터뷰 (2-3개 선택형)
       → 인터뷰 반영 2차 결과 (즉시)
       → "XX은 뭐라고 할까?" (호기심 트리거, 자동 진행)
       → 최종 정리 + Deliverables + DQ Score
  ```
- **핵심 변경:**
  - 1차 결과가 인터뷰 전에 나옴
  - 판단자가 인터뷰 Q1에서 동적 결정
  - AskUserQuestion으로 선택형 UI
  - Context Contract 유지 (개별 스킬 chaining용)
  - DQ Score는 선택적으로 제공

#### `/reframe` SKILL.md — 간소화
- **Before:** 561줄, 인터뷰 3개 + 코드블록 패턴 + 학술 인용 + 오케스트라 메타포
- **After:** ~290줄
- **핵심 변경:**
  - 즉시 초안 (Step 1) 추가 — 인터뷰 전에 결과 먼저
  - 인터뷰 3개 → 2개 (성격 + 핵심 걱정)
  - 코드블록+숫자 → AskUserQuestion
  - 전제 평가를 AskUserQuestion 하나로 번들링 (개별 질문 아닌 일괄)
  - "악보 해석" → "문제 재정의"
  - 학술 인용 제거 (rendering rules에서)
  - Adaptive rules, blind spot, journal은 유지

#### `/recast` SKILL.md — 간소화
- **Before:** 452줄, context extraction 상세 + build/decide 분기
- **After:** ~230줄
- **핵심 변경:**
  - 즉시 결과 (governing idea + structure → 바로 출력)
  - 핵심 로직 보존 (actor assignment 4-question, build/decide 분기)
  - "편곡" → "실행 설계"
  - reflection block 제거 (progressive flow에서 불필요)

#### `/rehearse` SKILL.md — 호기심 트리거 전환
- **Before:** 394줄, 페르소나 설정 UI + reflection block + 상세 review format
- **After:** ~210줄
- **핵심 변경:**
  - 페르소나 자동 생성 (사용자 설정 단계 제거)
  - "XX은 뭐라고 할까?" 프레이밍으로 진입
  - Devil's Advocate 유지 (핵심 가치)
  - 3가지 리스크 분류 유지 (critical/manageable/unspoken)
  - backward recommendation 유지
  - "리허설" → "사전 검증"

#### `/refine` SKILL.md — 간소화
- **Before:** 275줄, 이슈 선택 + 사용자 확인 + 상세 convergence
- **After:** ~150줄
- **핵심 변경:**
  - 이슈 선택/확인 단계 제거 → 자동으로 critical 먼저 수정
  - 결과만 보여줌 (변경 전/후)
  - convergence 로직 보존
  - "합주 연습" → "수정 반영"

#### 공통 변경사항
- **AskUserQuestion 도구 사용:** 모든 인터뷰 질문이 화살표 선택 UI로
- **교육→권위 톤:** "숨겨진 전제를 찾아보세요" → "이건 맞는데, 판단자는 여기서 물어볼 거다"
- **학술 인용 제거:** 사용자 대면 출력에서 Harvard/Stanford 등 제거
- **한국어 레이블 변경:** 오케스트라 한국어 레이블 → 직관적 표현
- **Context Contract 유지:** 개별 스킬 chaining 구조는 그대로

#### `/orchestrate` 정리
- **Before:** 154줄, /recast의 경량 중복
- **After:** ~15줄, /recast로 리다이렉트

#### `/help` 업데이트
- 새로운 description과 flow 반영
- "Decision Harness for AI" → "막막할 때 쓰는 구조화 도구"

## 7. 핵심 인사이트 (나중에 참고)

### 종화 피드백에서 가장 중요했던 한 문장
> "사람은 배우고 싶어하지 않아함. 생각은 내가 다 해봤으니까 너는 그냥 따라오면 돼."

이것이 Overture의 톤 전환 전체를 결정했다.

### 제품 변경에서 가장 중요했던 원칙
> "1차 결과가 인터뷰 전에 나온다."

기존: 인터뷰 → 분석 → 결과 (사용자가 기다림)
변경: 결과(초안) → 인터뷰 → 결과(업데이트) (사용자가 즉시 가치를 봄)

같은 기능인데 순서만 바꿨을 뿐이지만, UX 체감은 완전히 다르다.

### AskUserQuestion 발견
이미 `allowed-tools`에 선언되어 있었는데 안 쓰고 있었다. 코드블록+숫자 타이핑 대신 화살표 선택 UI를 제공할 수 있었던 것. SKILL.md 인스트럭션에 "AskUserQuestion 도구를 사용하라"고 명시하면 Claude가 해당 도구를 호출한다.

### "좁히라"는 마케팅 레벨이다
제품 기능 자체를 "개발자 기획 도구"로 제한하라는 게 아님. 4R 엔진은 범용이되, **첫 번째 스토리와 진입점**을 좁히라는 것. Notion이 "개인 노트 앱"으로 시작해서 "팀 운영 OS"가 된 것과 같은 경로.
