# Overture — Product Philosophy & Musical DNA (v2)

> *"실행 비용이 0에 가까워질수록, 실행 이전의 판단의 가치는 올라간다."*

## 0. 이 문서의 목적

이 문서는 Overture의 제품 철학을 정의합니다. 단순한 브랜딩 가이드가 아니라, **왜 이 도구가 존재하는지**, **어떤 경험을 만들고 싶은지**, 그리고 **오케스트라라는 메타포가 어떻게 제품의 모든 결정에 방향을 주는지**를 설명합니다.

README, 온보딩 카피, UI 마이크로카피, 기능 네이밍, 릴리즈 노트, LinkedIn 에세이에 이르기까지 — 이 문서가 모든 언어적·경험적 결정의 근거가 됩니다.

## 1. 핵심 철학: 왜 Overture인가

### 1.1 음악에서 가져온 본질

서곡(Overture)은 400년에 걸쳐 진화하면서, 단순한 기능에서 예술로, 그리고 다시 새로운 장르의 모태로 변화해왔습니다. 이 진화의 각 단계가 Overture라는 제품의 존재 이유를 설명합니다.

| 서곡의 역할 | 시대 | Overture에서의 대응 |
|---|---|---|
| **정숙시키기** — 관객의 주의를 집중시킴 | 1607, 몬테베르디의 토카타 | 흩어진 생각을 모으고, 과제의 본질에 집중하게 함 |
| **예고하기** — 본편의 주제를 미리 들려줌 | 1780s, 모차르트의 주제적 예고 | AI에게 넘기기 전, 결과물의 윤곽을 먼저 그리게 함 |
| **독립하기** — 서곡 자체가 하나의 완결된 작품이 됨 | 1826, 멘델스존의 연주회용 서곡 | 사고 과정 자체가 가치 있는 산출물이 됨 |

이 세 단계가 사용자 경험의 여정이기도 합니다. 처음에는 "AI 작업 전에 정리하는 도구"로 쓰기 시작하지만, 점차 "이 정리 과정 자체가 가장 중요한 작업"이라는 것을 깨닫게 됩니다.

### 1.2 한 문장 정의

**Overture — "Think before you recast."**

AI 에이전트가 실행하기 전에, 무엇을 왜 어떤 순서로 할지를 설계하는 도구.

### 1.3 제품이 믿는 것

- **악보 없는 오케스트라는 소음이다.** AI 에이전트가 아무리 뛰어나도, 무엇을 시킬지 정리되지 않으면 결과물은 소음에 가깝습니다.
- **지휘자는 악기를 연주하지 않지만, 악보를 읽을 줄 알아야 한다.** 비개발자도 AI 작업의 설계자가 될 수 있습니다. 코드를 쓸 필요 없이, 사고를 구조화할 수만 있다면.
- **좋은 서곡은 그 자체로 아름답다.** Overture로 정리한 문제 재정의, 실행 설계, 이해관계자 시뮬레이션은 AI를 쓰든 안 쓰든 그 자체로 가치 있는 산출물입니다.
- **서곡은 끝이 아니라 시작이다.** Overture는 결론을 내려주지 않습니다. 더 나은 질문을 던지게 합니다.

## 2. 본편의 구조: 악보 해석 → 편곡 → 리허설 → 합주 연습 → 공연

Overture의 핵심 플로우는 오케스트라가 공연을 준비하는 과정과 정확히 대응합니다. 이것은 비유가 아니라 설계 원리입니다.

### 2.1 전체 구조

```
┌─────────────────────────────────────────────────────────────┐
│                        OVERTURE                              │
│                "Think before you recast"                 │
│                                                              │
│   1. 악보 해석 ─── 문제 재정의                                │
│      Score Reading    과제의 진짜 질문 찾기                    │
│                                                              │
│   2. 편곡 ───────── 실행 설계                                 │
│      Arrangement      누가 어떤 역할을 어떤 순서로             │
│                                                              │
│   3. 리허설 ──────── 사전 검증                                │
│      Rehearsal        이해관계자 앞에서 한 번 연주해보기        │
│                                                              │
│   4. 합주 연습 ───── 피드백 반영                               │
│      Ensemble         맞을 때까지 반복하기                     │
│                                                              │
│   ─── 공연 (Output) ─────────────────────────────────────    │
│       Performance     프로그램 노트 / 파트보 / 총보 / 셋리스트  │
│                                                              │
│   ─── 조율 (독립 유틸리티) ──────────────────────────────     │
│       Tuning          여러 AI 결과물 비교 시 사용              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 맥락 누적 체인 — Overture의 해자

이 4단계는 단순한 순서가 아닙니다. 각 단계의 산출물이 다음 단계의 입력이 되면서, 맥락이 축적됩니다. 이 체인이 Overture의 경쟁 우위입니다.

```
악보 해석의 숨겨진 전제 → 편곡의 핵심 가정 → 리허설에서 가정 공격 → 합주 연습에서 우선 해결
악보 해석의 가설      → 편곡의 governing idea → 리허설에서 프리모템 → 합주 연습에서 더블루프 체크
```

오케스트라에서 악보 해석 없이 편곡할 수 없고, 편곡 없이 리허설할 수 없듯 — 사고의 깊이는 이전 단계의 축적 위에서만 가능합니다.

### 2.3 각 단계의 음악적 의미와 제품 경험

---

**1. 악보 해석 (Score Reading) → 문제 재정의**

오케스트라에서 악보 해석은 단순히 음표를 읽는 것이 아닙니다. 작곡가의 의도를 파악하고, 시대적 맥락을 이해하며, "이 곡이 정말 말하고 싶은 것이 무엇인가"를 찾는 과정입니다. 같은 악보를 카라얀과 번스타인이 전혀 다르게 해석하듯, 같은 과제도 해석에 따라 완전히 다른 방향이 됩니다.

Overture에서 악보 해석은 주어진 과제를 그대로 받아들이지 않고, 그 뒤에 숨겨진 진짜 질문을 찾아내는 과정입니다.

**내장된 사고 패턴 (Overture Method):**
- 가설 기반 사고: "이 과제의 진짜 이유는 X, 풀어야 할 질문은 Y"
- 리프레이밍: 대안적 프레이밍 2-3개 생성
- 4축 전제 점검: 고객 가치 / 실행 가능성 / 사업성 / 조직 역량
- 이슈 트리: MECE 분해

**UX 원칙:**
- 이 단계에서는 속도보다 깊이가 중요합니다. 사용자가 "아, 내가 진짜 해결해야 할 문제가 이거였구나"라고 느끼는 순간을 만들어야 합니다.
- 대화형 모드(Interview)가 기본값인 이유도 이것입니다. 질문을 하나씩 던져주면서 사용자 스스로 생각을 펼치게 유도합니다.

**마이크로카피 톤:** 질문형, 탐색적.
- "이 과제가 정말로 묻고 있는 것은 무엇입니까?"
- "성공했을 때, 무엇이 달라져 있나요?"
- "이 과제에 숨겨진 전제가 있지 않습니까?"

---

**2. 편곡 (Arrangement) → 실행 설계**

악보 해석이 "이 곡이 무엇을 말하는가"를 이해하는 과정이라면, 편곡은 "이것을 어떤 악기로, 어떤 배치로 연주할 것인가"를 결정하는 과정입니다. 같은 멜로디도 현악 4중주로 편곡하느냐, 풀 오케스트라로 편곡하느냐에 따라 완전히 다른 색채를 띱니다. 어떤 악기가 언제 들어오고, 어떤 순서로 겹쳐지는지 — 이것이 편곡의 핵심입니다.

**이 단계가 Overture에서 가장 중요합니다.** 여기서 과제의 전체 구조가 결정됩니다.

**내장된 사고 패턴 (Overture Method):**
- 피라미드 원칙: governing idea (한 문장 핵심 방향)
- SCR 스토리라인: 상황(Situation) → 문제(Complication) → 해결(Resolution) 서사 구조
- 기대 산출물: 각 단계가 "뭘 만드는지" 구체적 명시
- 판단 포인트: 사람이 결정할 것 + 선택지 + 트레이드오프
- 핵심 가정: importance × certainty 매트릭스 + if_wrong(틀렸을 때의 결과)
- 크리티컬 패스: 병목 단계 식별

**UX 원칙:**
- 시간축과 역할 배분이 동시에 보여야 합니다. 총보(full score)처럼 — 각 파트가 언제 연주하고 언제 쉬는지 한눈에.
- WorkflowGraph를 통해 카드 기반으로 워크플로우를 시각화합니다.
- maxTokens 3500으로 가장 큰 출력을 생성하는 단계이니, 로딩 경험이 특히 중요합니다.

**마이크로카피 톤:** 구조적, 설계적.
- "이 과제의 한 문장 핵심 방향은 무엇입니까?"
- "이 결과물은 누구에게 넘겨지나요?"
- "사람이 반드시 판단해야 하는 지점은 어디입니까?"

---

**3. 리허설 (Rehearsal) → 사전 검증**

오케스트라의 리허설은 본 공연 전에 실제로 한번 연주해보는 것입니다. 악보 위에서는 완벽해 보이던 편곡이, 실제로 소리를 내보면 문제가 드러납니다. 호른의 음역이 너무 높다든지, 두 파트의 리듬이 충돌한다든지 — 리허설은 "계획의 약점을 미리 발견하는" 과정입니다.

Overture에서 리허설은 이해관계자의 시점에서 실행 설계를 사전 검증하는 단계입니다. 편곡이 아무리 정교해도, 실제로 이해관계자 앞에 내놓았을 때 어떤 반응이 나올지 미리 시뮬레이션해봐야 합니다.

**내장된 사고 패턴 (Overture Method):**
- 프리모템 (Gary Klein): "이 계획이 이미 실패했다면, 이유는?"
- 리스크 3분류: 🔴 핵심 위협(critical) / 🟡 과장된 우려(manageable) / 🟣 침묵의 리스크(unspoken)
- 가정 공격: 편곡의 핵심 가정 중 검증 안 된 것 집중
- 승인 조건: "이걸 보여주면 OK하겠다" 구체적 조건

**UX 원칙:**
- 페르소나가 살아있는 이해관계자처럼 느껴져야 합니다. influence 레벨(high/medium/low)에 따라 피드백의 강도가 달라집니다.
- 리스크 3분류가 시각적으로 즉시 구분되어야 합니다. 🔴/🟡/🟣 색상 코딩.
- failure_scenario, untested_assumptions, approval_conditions가 구체적으로 출력됩니다.

**마이크로카피 톤:** 도전적, 시뮬레이션적.
- "CFO의 첫 반응을 예측합니다"
- "이 계획이 이미 실패했다면, 어떤 이유 때문이었습니까?"
- "아무도 말하지 않지만 모두가 알고 있는 리스크는 무엇입니까?"

---

**4. 합주 연습 (Ensemble Practice) → 피드백 반영**

리허설에서 문제가 발견되면, 합주 연습이 시작됩니다. 오케스트라에서 합주 연습은 리허설의 피드백을 반영하여 다시 맞춰보고, 각 파트 사이의 밸런스를 조율하며, 전체가 하나의 소리로 수렴해가는 과정입니다. 한 번에 완벽해지는 법은 없으므로, "맞을 때까지 반복"합니다.

**내장된 사고 패턴 (Overture Method):**
- PDCA: 수정 → 반영 → 재검증 → 표준화
- 더블루프 러닝: 지적받은 걸 고치는 것뿐 아니라 "애초에 질문이 틀렸나?"까지
- 하모니 지수: 수렴 정도 추적

**UX 원칙:**
- "하모니 지수"를 통해 수렴 정도를 시각적으로 보여줍니다. 각 이해관계자의 피드백이 얼마나 반영되었는지, 아직 미해결인 쟁점이 무엇인지.
- 리허설 → 합주 연습 → (필요 시) 다시 리허설로 돌아가는 루프가 자연스러워야 합니다.
- "하모니 완성" 상태에 도달하면 공연(Output)으로 넘어갈 수 있습니다.

**마이크로카피 톤:** 반복적, 수렴적.
- "리허설 피드백 중 미반영 항목이 있습니다"
- "핵심 위협 2건이 아직 해결되지 않았습니다"
- "하모니 지수: 78% — 합주를 한 번 더 하시겠습니까?"
- "하모니가 완성되었습니다. 공연을 준비할까요?"

---

**→ 공연 (Performance) → 산출물 4종**

모든 준비가 끝나면 공연입니다. 오케스트라의 공연은 관객에게 전달되는 최종 결과물이지만, 그 뒤에는 악보 해석, 편곡, 리허설, 합주 연습의 모든 과정이 축적되어 있습니다.

Overture의 공연은 4종의 산출물로 구현됩니다. 각 산출물은 서로 다른 청중과 용도를 가집니다.

| 비유명 | 영문명 | 청중과 용도 | 오케스트라 대응 |
|--------|--------|------------|----------------|
| **프로그램 노트** | Project Brief | 경영진/팀에 공유하는 의사결정 기록 | 관객이 읽는 프로그램 노트 — 곡의 배경과 핵심 메시지 |
| **파트보** | Prompt Chain | AI에 순서대로 입력할 프롬프트 세트 | 각 연주자가 받는 개별 파트보 — 자기 역할만 명확히 |
| **총보** | Agent Spec | LangGraph/CrewAI 구현 출발점 (YAML) | 지휘자가 보는 총보 — 모든 파트가 동시에 보임 |
| **셋리스트** | Execution Checklist | 순서대로 체크하며 실행하는 리스트 | 공연 당일의 셋리스트 — 순서와 타이밍 |

---

**조율 (Tuning) → 독립 유틸리티**

오케스트라에서 조율(tuning)은 공연 전에 모든 악기의 음높이를 맞추는 과정입니다. 오보에가 A4(440Hz)를 불면 나머지 악기가 그에 맞춥니다. 본편 플로우의 일부가 아니라, 필요할 때 언제든 할 수 있는 독립적인 행위입니다.

Overture에서 조율은 여러 AI 결과물을 나란히 놓고 비교·합성하는 유틸리티입니다. 메인 플로우(악보 해석→편곡→리허설→합주 연습)에서는 빠져 있지만, 편곡 단계에서 여러 AI의 출력을 비교해야 할 때, 또는 리허설 후 대안적 접근을 검토할 때 독립적으로 사용합니다.

## 3. 네 가지 터치포인트에서의 경험 설계

### 3.1 첫 인상 — 토카타(Toccata)

몬테베르디의 「오르페오」는 9마디의 짧은 토카타로 시작합니다. 화려하지 않지만, "지금부터 무언가가 시작된다"는 신호를 분명하게 보냅니다. 17세기에 이 9마디는 세 번 반복 연주되었는데, 그것만으로 관객은 자리에 앉고, 잡담을 멈추고, 무대에 시선을 모았습니다.

Overture를 처음 열었을 때의 경험이 이래야 합니다.

**구현 원칙:**
- **비어있지만 비어있지 않은 화면.** 대시보드에 카드와 버튼이 가득한 게 아니라, 하나의 질문이 중심에 있어야 합니다. "아직 서곡이 시작되지 않았습니다"라는 빈 상태 메시지가 그 자체로 초대입니다.
- **첫 액션까지의 거리를 최소화.** 온보딩 튜토리얼 5단계가 아니라, 과제를 한 문장 입력하면 바로 악보 해석이 시작되는 경험. 토카타가 9마디인 것처럼 — 짧지만 분명하게.
- **대화형 모드(Interview)가 기본값.** 처음 온 사용자에게 빈 텍스트박스를 던지지 않습니다. 질문을 하나씩 던져주면서, 자연스럽게 사고를 펼치도록 안내합니다.

**온보딩 내러티브 (초안):**

```
Overture에 오신 것을 환영합니다.

AI에게 일을 시키는 건 쉬워졌습니다.
어려운 건 무슨 일을 시킬지 정하는 겁니다.

악기가 아무리 좋아도, 악보 없이는 음악이 되지 않습니다.
Overture는 실행 전의 판단을 구조화하는 도구입니다.

과제를 입력하면, 네 번의 연습을 거쳐 공연 준비를 마칩니다.
악보 해석 → 편곡 → 리허설 → 합주 연습 → 공연.

자, 첫 번째 악보를 펼쳐볼까요?
```

### 3.2 사고의 흐름 — 맥락 누적의 감각

서곡 역사에서 결정적인 전환은 글루크의 개혁(1767)이었습니다. "서곡은 드라마와 연결되어야 한다"는 선언 — 서곡의 마지막 음이 끊기지 않고 첫째 막의 음악으로 흘러들어가는 경험. 이 끊김 없는 연결이 Overture의 단계 전환에서 일어나야 합니다.

**구현 원칙:**
- 악보 해석에서 발견한 "숨겨진 전제"가 편곡의 "핵심 가정" 필드에 자동으로 넘어가는 것을 사용자가 **눈으로 확인**할 수 있어야 합니다.
- 단계 완료 시 다음 단계로의 전환 메시지가 맥락 연결을 보여줍니다:
  - 악보 해석 → 편곡: "주제가 잡혔습니다. 이제 각 파트에 역할을 배정합니다."
  - 편곡 → 리허설: "악보가 완성되었습니다. 이해관계자 앞에서 연주해봅니다."
  - 리허설 → 합주 연습: "첫 합주가 끝났습니다. 피드백을 반영하여 다시 맞춰봅니다."
  - 합주 연습 → 공연: "하모니가 완성되었습니다. 무대에 올릴 준비가 되었습니다."
- 어떤 단계에서든 이전 단계로 돌아갈 수 있되, 돌아가면 그 이후 단계에도 영향이 전파됨을 보여줍니다. "악보를 수정하면 편곡도 다시 해야 할 수 있습니다."

### 3.3 AI 결과물의 순간 — 주제적 예고의 실현

모차르트의 「돈 조반니」 서곡이 위대한 이유는, 서곡에서 들린 테마가 오페라 본편에서 다시 등장할 때 관객이 "아, 저 멜로디!"라고 느끼기 때문입니다. 예고된 것이 실현되는 순간의 감동.

**구현 원칙:**
- 공연(Output) 생성 시, 사용자가 **악보 해석에서 정의한 가설이나 핵심 질문이 최종 산출물에 어떻게 반영되었는지** 추적할 수 있어야 합니다.
- 프로그램 노트(Project Brief)에는 전체 사고 과정의 궤적이 담깁니다 — "처음에 이런 가설로 시작했는데, 리허설에서 이런 리스크가 발견되어, 합주 연습을 통해 이렇게 수정되었다"는 서사.
- 파트보(Prompt Chain)의 각 프롬프트에는, 그 프롬프트가 편곡의 어떤 단계에서 왜 이런 순서로 배치되었는지의 근거가 주석으로 달립니다.

### 3.4 작업 후 성찰 — 코다(Coda)와 메타 튜닝

음악의 코다는 곡의 끝에 붙는 종결부로, 전체를 되돌아보고 마무리 짓는 구간입니다. 때로는 전체 곡에서 가장 인상적인 순간이 코다에서 일어나기도 합니다. 베토벤의 교향곡 5번 4악장 코다가 그렇습니다 — 끝이라고 생각했던 순간에, 가장 강렬한 확인이 옵니다.

멘델스존이 서곡을 오페라에서 분리시켜 독립적인 예술 작품으로 만든 것은 음악사의 결정적 전환이었습니다. 서곡이 더 이상 "본편을 위한 준비"가 아니라, 그 자체로 완결된 가치를 갖게 된 것.

**구현 원칙:**
- 합주 연습이 끝나고 공연(Output)을 생성한 후, 부드럽게 성찰을 제안합니다. 강제가 아니라 제안.
- 성찰 질문 세 가지:
  1. "처음 악보를 받았을 때의 이해와 지금의 이해가 어떻게 달라졌습니까?"
  2. "이 과정에서 가장 놀라운 발견은 무엇이었습니까?"
  3. "다음에 비슷한 과제를 만나면 무엇을 다르게 하겠습니까?"
- 이것이 **메타 튜닝**의 실제 적용입니다. AI의 출력을 그대로 소비하는 것이 아니라, AI와의 협업 과정에서 자신의 사고방식 자체를 조율(tuning)하는 것. 합주 연습에서의 더블루프 러닝이 여기까지 이어집니다.

### 4.2 마이크로카피 가이드

**로딩 메시지:**
- 악보 해석: "악보를 펼치고 있습니다..."
- 편곡: "각 파트에 역할을 배정하고 있습니다..."
- 리허설: "이해관계자를 무대 앞으로 초대하고 있습니다..."
- 합주 연습: "각 파트의 소리를 맞추고 있습니다..."
- 공연: "무대 조명이 켜지고 있습니다..."

**빈 상태:**
- 워크스페이스: "아직 서곡이 시작되지 않았습니다."
- 특정 단계: "무대에 연주자가 없습니다."
- 공연 탭: "아직 공연 준비가 완료되지 않았습니다."

**단계 전환:**
- "주제가 잡혔습니다." (악보 해석 완료)
- "악보가 완성되었습니다." (편곡 완료)
- "첫 합주가 끝났습니다." (리허설 완료)
- "하모니가 완성되었습니다." (합주 연습 완료)

**에러/경고:**
- "이 악장의 핵심 모티프가 아직 정의되지 않았습니다."
- "편곡의 핵심 가정이 리허설에서 공격받지 않았습니다. 계속하시겠습니까?"

**저장/내보내기:**
- "이 서곡의 총보를 내보냅니다"
- "프로그램 노트를 공유합니다"

**돌아오기:**
- "미완의 서곡을 이어 쓰기"

### 4.3 비유의 원칙: 이해를 돕되 가리지 않게

오케스트라 메타포는 **업무명과 항상 병기**합니다. 이것은 핵심 설계 원칙 4번입니다.

- ✅ "악보 해석 — 문제 재정의"
- ✅ "리허설 — 사전 검증"
- ❌ "악보 해석" (업무명 없이 비유만)

음악을 모르는 사람도 "문제 재정의"를 보고 이해할 수 있어야 합니다. 음악을 아는 사람은 "악보 해석"에서 더 깊은 공감을 느끼되, 모르는 사람이 소외되지 않는 수준.

**한 문장에 오케스트라 비유 두 개 이상 쓰지 않습니다.**
- ❌ "악보 해석에서 정의한 모티프를 리허설의 프리모템으로 공격하여 합주 연습에서 하모니를 맞춥니다."
- ✅ "1단계에서 세운 가설이 3단계에서 어떻게 검증되는지 확인합니다."

## 6. 디자인 무드: 새벽의 콘서트홀

### 6.1 시각적 방향

"AI 생산성 도구"의 전형적인 디자인 — 대시보드, 카드, 보라색 그래디언트 — 에서 벗어납니다.

Overture의 시각적 참조는 **새벽 시간의 빈 콘서트홀**입니다. 공연 전의 고요함. 악보대 위에 총보가 펼쳐져 있고, 객석은 비어 있지만 곧 무언가가 시작될 것이라는 기대감.

- **컬러**: 따뜻한 아이보리/크림 바탕 + 진한 다크 네이비/차콜 텍스트 + 골드/앰버 악센트. 차가운 테크 블루를 피합니다.
- **타이포그래피**: 본문은 읽기 편한 세리프 느낌. 단계 제목은 클래식한 세리프. 코드나 수치는 모노스페이스. "책을 읽는 느낌".
- **여백**: 과감하게. 콘서트홀의 빈 공간이 음향을 풍부하게 하듯, 화면의 여백이 사고를 풍부하게 합니다.
- **모션**: 최소한, 느리고 우아하게. 스프링 기반 트랜지션(cubic-bezier(0.16, 1, 0.3, 1)). body::after 노이즈 오버레이로 미세한 질감.
- **리스크 색상 체계**: 🔴 #E24B4A(핵심 위협) / 🟡 #EF9F27(과장된 우려) / 🟣 #7F77DD(침묵의 리스크)

### 6.2 사운드 (선택적, 향후 고려)

- **단계 전환 시**: 아주 짧은 피아노 한 음(A4=440Hz). 오케스트라 조율의 기준음. 선택적 기능.
- **하모니 완성 시**: 부드러운 화음의 아르페지오. 해결(resolution)의 느낌.
- **기본값은 무음.** 설정에서 켜는 옵션.

## 7. 경쟁 프레이밍

### 7.1 시장 내 포지셔닝

```
                  판단(Judgment) 계층
                        ▲
              ┌─────────┤
              │ Overture │  ← 여기
              └─────────┘
    ────────────────────┼──────────── 인간과 기계의 경계
              ┌─────────┤
              │ LangGraph│
              │ CrewAI   │  실행(Execution) 계층
              │ AutoGen  │
              └─────────┘
                        ▼
```

### 7.2 경쟁하지 않는 것들

- LangGraph, CrewAI, AutoGen → 실행 인프라. Overture의 **아래** 계층. 보완 관계.
- Notion AI, ChatGPT → 범용 AI 어시스턴트. Overture는 "과제를 AI에게 넘기기 **전**" 단계에 특화.
- 총보(Agent Spec) Output이 LangGraph/CrewAI 구현의 출발점이 된다는 점에서, 실행 계층과 자연스럽게 연결.

## 10. 이 제품이 만들고 싶은 순간

Overture가 성공했다면, 사용자는 다섯 가지 순간을 경험합니다:

1. **"아, 내가 진짜 풀어야 할 문제가 이거였구나."** — 악보 해석에서, 원래 받은 과제와 진짜 질문이 다르다는 것을 깨달을 때.

2. **"이렇게 배치하니까 전체가 보인다."** — 편곡에서, governing idea 한 문장이 잡히고 전체 워크플로우가 구조화되는 순간.

3. **"이 사람이 이 부분을 찌르겠구나."** — 리허설에서, 미처 생각 못한 침묵의 리스크(🟣)를 발견할 때.

4. **"맞을 때까지 맞추니까 진짜 단단해졌다."** — 합주 연습에서, 하모니 지수가 올라가면서 계획의 빈틈이 메워지는 것을 볼 때.

5. **"이 과정 자체가, AI 결과물보다 더 가치 있었네."** — 공연 후, 전체 사고의 변화를 돌아보며 메타 튜닝이 일어날 때.

이 다섯 순간이 Overture의 존재 이유입니다.
